# BidWorks MVP 平台全栈集成联调测试计划表 (Integration Test Plan)

本联调计划涵盖一至五迭代全部串联的 P0 核心业务，确保没有因各迭代单项开发而产生的数据脱钩、状态闭锁和越权漏洞。

---

## 一、集成测试目标

说明本迭代要解决什么问题：
- 验证 BidWorks 投标一体化生产平台在首个开发周期（迭代 1-5）结束后，各个核心业务子系统在物理网络和本地进程中的联调稳定性、数据无损跨表交互。
- 确认全案业务核心场景：从无招标书速建、招标PDF/Docx物理提取切片、AI 多项商务或资质点精准提取（附带页码定位对照）、后向自左向右时间表自动铺设倒排、专岗位专属隔离个人工作台 (Personal Workbench) 协作空间、提报送审前 Live 扫描自检及残留徐汇废案屏蔽词卡提交、直至领导划段提 ACTIVE 批注、主管 Resolved、主数据重大变更级联需复核状态提醒与全岗位手动复核解除提醒的完整 P0 天堑流程是否无阻跑通。
- 保证系统安全红线稳定在生产环境。

## 二、端到端测试场景 (E2E Test Scenarios)

### 场景 1：有招标书上传创建项目及托管流程 (Has-Tender Setup Flow)
* **前置条件**：系统正常初始启动，测试物理文件服务器（Storage）空间充足，多角色初始测试账号就绪。
* **测试账号角色**：营业专员 (BD/Commercial - 张三)
* **测试数据**：一个包含文字型、大小为 4MB、内容清晰真实的日资精密模具工程招标书物理 docx 模板。
* **操作步骤**：
  1. 张三打开 BidWorks 登陆页，输入张三账户，进入主项目大盘。
  2. 点击右上角“建立投标项目”大浮层，挑选“读取招标文书建项”大标签。
  3. 拖拽准备好的本地 4MB 标书文件到上传区上。
  4. 输入工程简称：“上海精密二期工程”。
  5. 点击确定提交。
* **预期结果**：
  - 前端：物理上传进度由 0% 平滑划归 100%，无渲染黑白死锁，上传框下方带有已托管存储提示。
  - 后端：本地物理存储服务器目录下正确生成规范更名后的物理文件：`doc-[随机ID].docx`，SQLite 对应的 `project_documents` 表新增一行，且其 `is_sensitive` 属性默认为 0，`is_ai_allowed` 被安全设为默认的 0。
* **验收标准**：文件的 md5 值在托管存储和本地原盘保持一致，没有传输断流、文件名乱码、或者因大小校验引发的溢出丢件。
* **失败处理建议**：如果遇到前端拖放后报 `413 Payload Too Large`，建议立即排查 Node 后端的 multipart bodyParser 上限参数是否由于拼写错误写成了字节偏小等，将大小上限设为 P0 阶段安全理智的 20MB。

### 场景 3：招标解析结果人工审定确认并同步项目主数据库 (Extraction & Review Integration)
* **前置条件**：场景一里由高密上传成功的精密二期招标文件 docx 已安全留置。
* **测试账号角色**：项目负责人 (BM - 李四)
* **测试数据**：物理文档 docx 中明写：“基地规划总占地总建筑面积为 90000 平方米，本期工程总历时 400 个日历天”。
* **操作步骤**：
  1. 李四以其 BM JWT 信息鉴权登入，点击项目一侧的“智能分析与对照”按钮。
  2. 弹出对照大卡，点击“开始提取主体参数”。等待菊花进度跑满。
  3. 解析完毕，右列推荐出的面积变量为 `90000`（原处附带页码：【引自第 4 页，第 2 行。原文：本期基地....共九万㎡】）；
  4. 李四发现由于追加答疑，工期写 400 天被写错，其在右侧输入框将 400 手动改为 `380 天`。
  5. 点击卡下醒目“核校无误，同步到项目主数据”大绿色键。
* **预期结果**：
  - 前面虚线灰色大标志 Pending 状态消退，全部高亮亮起。
  - 去数据库中查询 `project_master_data` 得到：建筑面积为 `90000`，工天数更新为 `380`，状态为 Confirmed，数据来源为 `人工校正同步`，且审计记录表 `master_data_change_logs` 完整留痕。
* **验收标准**：通过人工覆写校准的 `380` 指标，必须准确同步覆写至主数据表。且凡是修改了的指标段，在右边能够顺畅触发左边第 4 页的定位高亮联动。
* **失败处理建议**：在多端解析时若报模型超时无法回字。系统统一截获并将连接超时参数强制调至最少 30 秒，并在前端呈优雅降级提示，警告可以手工从容填单。

### 场景 4：资料上传、自检验、提交送审核流 (Upload, Self-Check, Submit Workflow)
* **前置条件**：李四已将 `t3` (施工大纲方案编制任务) 指派给施工专员陈七。且系统敏感黑名库预置了“徐汇”这个历史废项目未脱密过滤地名词。
* **测试账号角色**：施工专员 (Tech Specialist - 陈七)
* **测试数据**：陈七准备了一分套皮老标书，里面含有文字：`“本方案比照青浦基地，施工地点在大同徐汇一期工程里，建筑大楼规划总建筑面积为 21000 平方米”`。
* **操作步骤**：
  1. 陈七利用专员账号登入个人工作台，点击受托的技术编制任务 t3 进入，拖入此套皮 Word 文件。
  2. 系统生成 v1 版。陈七点击 “提报前一致性自检测” 大按钮。
  3. 闪闪自检弹出。红字提示两个致命报警：
     - **一致性错**：目前项目主数据建筑面积是 90000 平方米，你的这本技术方案第 2 页写成了 21000，严重不吻合。
     - **泄密错**：第 11 页面中发现了未脱密的历史非本工程地名旧敏感词 “徐汇”，红色最高拦截警告，不可忽略。
  4. 陈七强行尝试在自检框下点忽略“徐汇”强行点击忽略，并输入：“这是微小的旧拼写没关系”。
  5. 再次点击“向领导钱八提交送审”。
* **预期结果**：
  - 后端直接卡绝。在陈七不改原 Word 剔除“徐汇”前，系统将其“提交送审”功能硬置灰卡死。后端接口在 `/self-check/ignore` 第一排就拦截 `403`。
  - 陈七在自己电脑上把 21000 改为 90000，删去徐汇两个字。重新上传为 v2 版本。
  - 再次启动智能自检，提示和需复核提醒全部消除。
  - 陈七一键轻松拿下完成 “提交送审”，任务状态由 DOING 平滑变为 `PENDING_REVIEW`（待审核）。
* **验收标准**：陈七如果强带“徐汇”敏感词被硬性打回。只要自测检出 FATAL 属性，系统不管收到何种忽略请求一律在后端硬卡截。
* **失败处理建议**：有些 Docx 解析时对特殊中括号不敏感找不到字词导致自检未命中。此时应核对切片后文书格式是否做了全包容性的纯字符流提纯降质转化，确保不漏网泄密字库。

### 场景 5：领导结构化审核意见发出、对话、消项 resolved 通关 (Review Comments Close Flow)
* **前置条件**：陈七成功以 v2 的干净方案送审，任务状态变为 `PENDING_REVIEW`。
* **测试账号角色**：审核主管 (Reviewer - 钱八)
* **测试数据**：批注要求说明：“第二页中关于施工起重机的起载规范有明显技术不当，要求更正。”，限定整改天：1日内完成。
* **操作步骤**：
  1. 审核主管钱八 JWT 登入专属领导工作盘，页面提示有任务 “陈七施工方案编制待审”。
  2. 点击进入阅览。他发现前面列出了：陈七在提交前，曾经自己做自查时，故意写字抗议豁免了面积有 1㎡ 小尾数误差的偏差陈述（陈述内容：本期由装配式剪裁所以有1平米尾差合理）。
  3. 钱八由于对起重安全不放心。选中第二页 “液压升降法” 文字段。
  4. 钱八点击提出“新增批注”。写入：由于安全性，液压在夏季有不妥，请在明天前改用更为平稳的三级滑块，并指责给陈七。
  5. 此时陈七尝试在自己的台面一键点击将方案标记为 “最终大底定标本 (Finalize)” 封存。
  6. 陈七在后面由于知道不能定稿，不得不修改 Word 原文重上第 3 版，并在钱八的批注贴纸下方对话框留言：“已经更正为了三级滑块并上载了第3版，请查验！”。
  7. 钱八收到陈七提示，对重上方案满意，钱八点击上面 “关闭该处审核意见（Resolved）”。
  8. 陈七再次到页面上尝试将其标记为“最终版（LATEST_APPROVED）”。
* **预期结果**：
  - 步骤 5 尝试最终定稿时：后端校验到 `review_comments` 仍有 1 项在库 active 状态批注，接口立马吐回 `{ "blockReason": "ACTIVE_COMMENTS_REMAINING" }` 拦截，最终版置灰，定标流被重创中止。
  - 步骤 7 钱八点击 Resolved 后，该批注物理状态更新为 RESOLVED。
  - 步骤 8 没有任何挂起在库批注，定稿大闸解除，最终方案成功标记通过，落底。
* **验收标准**：全闭环，领导 ACTIVE 批注为零是通关前置物理死约束，非 Reviewer 无法越权消 resolved。
* **失败处理建议**：如果点击 “标记最终版” 报 SQL 语法错误，立刻排查 SQLite 关联子表的 count 函数调用参数，确保传入正规的 documents FK 关联。

### 场景 6：主数据重大更迭高风险触发下游级联状态打回、手动复核解除需复核状态 (Cascading Impact Flow)
* **前置条件**：项目已处于封阶段尾声，多岗位（王五设计、陈七施工等）都已经提报且审核领导 Resolved，控制台正在汇总定。
* **测试账号角色**：营业专员 (BD - 张三)、设计人员 (王五)
* **测试数据**：张三由于发包方突然发来第二轮修改合并，把项目主数据中的 grossFloorArea（建筑面积）由 `90000` 强推改为了 `105000 ㎡` (重要特征中高风险变动)。
* **操作步骤**：
  1. 张三对该段进行 PUT 修改。
  2. 系统底盘运算。
  3. 设计专员王五（此前已 Accepted 通合）登录工作台。
  4. 页面正中间瞬间顶出遮罩一排红色大字：“【主数据发生重大变动 ── 相关方案编制被迫降重处理风险评估单】”。
  5. 评估卡以极漂亮高对比展示：主修改面积由原：90000㎡，变现：105000㎡，差值长了1.5万。
  6. 此时李四在主管控制台一键点 “标记最终本 Finalize”，直接被限制卡闸拦截。
  7. 王五被迫在其电脑处将 105000 新数据合入文档中重编。
  8. 重新丢入平台 v5。
  9. 点击一键复核一键确认：“本人王五，已针对本次1.5万㎡大更完成了设计文本再纠正”。
* **预期结果**：
  - 步骤 1 张三 PUT 的瞬间，底层级联变更提醒影响：设计任务、施工方案其 status 被全数归洗设置为 `NEED_REWORK`（置需复核状态）。
  - 步骤 6 李四定标动作由于还有 `NEED_REWORK` 状态，强遭物理阻绝不可定。
  - 步骤 9 王五以及其关联施工陈七全部完成了复核手动清零。需复核状态提醒清除。
  - 李四的总控定稿闸顺利点亮放行。
* **验收标准**：低风险变（如甲方更名，拼写字眼）完全不联动下游做状态降级，但中高重大指标一更，下游执行自动打回需复核状态重审，且直至 100% 卡口解开强力阻赛。
* **失败处理建议**：对于高敏时序不匹配极易在多岗位协作时产生数据库回滚。开发在级联修改各岗位卡状态时，务必将它和主数据修改在一个 SQL Transactions 包，并采用同步分发。

## 三、安全测试用例 (Security Test Cases)

### 1. 普通登录用户跨单越权物理封杀测试
- **目的**：确保非本项目名专责人、非管理员无法嗅探、非法读取或跨案注入 PUT 修改。
- **前置配置**：项目 P1（只指配给了王五，陈七在 P1 无归属）。
- **步骤**：使用陈七的 API Bearer Bearer 绕过前端，直接用 POST 工具指向 `/api/projects/p1/master-data` 进行纂改。
- **验证通过标准**：后端安全中间件极速抓黑：`[IP地址] 账号陈七跨单尝试写入不归属项目P1的主 facts 事实数据，被系统拦截阻绝并发出 403 Forbidden`。安全在案没有任何漏。

### 2. 精炼报价资料商业秘密隔离测试
- **目的**：防止工程中高保密极敏感的“工程预算一键底卡”方案被非授权（如普通设计员、或是施工图编制人）私自下载窃取。
- **前置配置**：任务卡 $t4$ 为工程施工技术底盘，任务卡 $t5$ 为钱八持有的高度机密的工程概算商务卡。
- **步骤**：使用设计员王五的 JWT 凭证，尝试拉取或下载 $t5$ (概算任务卡) 下面的 doc 物理文档。
- **验证通过标准**：系统查实王五的职位和指派归属不对应，拒绝此下载文件写操作返回 403，安全。

### 3. 高密涉敏资料物理隔离不网送测试
- **目的**：保证涉敏文书在 `isSensitive=1` 默认不可调 AI 模型，防止泄密事故。
- **前置配置**：将文件 "某密项目招标文件.pdf" 状态标记为 `isSensitive = 1`, `isAiAllowed = 0`。
- **步骤**：李四点击对该文件的 “智能对照解析” 大动作。
- **验证通过标准**：网关检测到文件是高危密件，在向外部公网大模型发送 API 通讯的第一秒，进程强行阻断并丢抛 `AI_SECURITY_VIOLATION`。

## 四、AI 专属功能测试用例 (AI Specialized Testing)

### 1. PDF 非幻觉提取对照完整测试
- **目的**：验证用适配层解析 PDF 标书，模型抽出的指标带有 sources 正确结构体，无常识拼写错。
- **预期结果**：

- **目的**：检验 Word 本地段提取切片投射可靠度。
- **预期结果**：
  - 各类不规则空括号能够多极被解析为段，AI 能将其拼装并向右栏呈列出潜在工历天限。

### 3. 涉敏强制物理跳过断网测试 (Sandbox Ai Non-readable)
- **目的**：验证高敏感文书发起 parse 直接退出。
- **验证通过标准**：`ai_token_usage_logs` 里本次调用记录 tokens 物理为零（说明彻底处于沙盘断网拦截态下，没有给外部漏出半点数据）。

## 五、失败处理建议汇总

1. **AI 服务彻底不通/物理挂死断网降级退路**：
   - *方案*：一旦检测到外部 API 响应高于 20s 或是返回 500。一律将该文件的 AI 解析列置为 `"当前模型繁忙，已降级启用纯手工速创"`，前端亮黄色警。
2. **多用户同时覆盖文件 SQLite 导致数据库发生 Lock 报错**：
   - *方案*：将 SQLite 采用 Write-Ahead Logging (WAL) 模式并在连接中把 timeout 延展长达 5000 毫秒，多岗位并发行。
3. **一致性检查里正则未命中数据误差大**：
   - *方案*：不对用户自检报警强制卡死。只要其手写理由符合 5 个字限制，统统一律放行，仅向审计本保存抗辩踪，不致于阻断赶快标编制正常。
