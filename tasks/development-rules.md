# BidWorks MVP 开发规范与纪律守则 (Development Rules)

本规范文件是 BidWorks MVP 全开发周期的指导红线，旨在约束开发边界、杜绝范围蔓延、保障系统安全性与工程健壮度。本期开始前和过程中的每个操作都必须对照执行。

---

## 一、范围控制规则

- **不得擅自扩大 P0 范围**：开发与系统设计必须完全束紧于本次任务拆分的 [P0] 范围内，不支持非 P0 自建、自创的扩。
- **不得加入 P1/P2 功能**：所有标注为 P1 或 P2 的功能（多极审批自由设定、跨公司比对甘特等），严禁出现在本开发分支任何源文件和表结构中。
- **不得加入自动报价**：报价卡片（工程概算）仅支持手动上传核对与基本记录，系统绝对不开发任何一键自动询价、智能工程主材自动报价的算法模型。
- **不得加入 BIM 深度联动**：不添加复杂的 CAD 工程图纸、BIM 三维渲染、或在线三维大件拆量功能，只做纯文本级资料及表格的匹配。
- **不得加入管理层大屏**：去掉任何不具备实际操作交互、带有空 Telemetry 状态和装酷性质的管理层炫目 LED 大屏。
- **不得加入移动端完整功能**：本产品第一优先级只解决 PC 浏览器端双栏、主控大物理页面的良好运作，不提供特种的原生 App 或复杂混合封装。
- **不得加入 Office 插件**：不编写 Word/Excel/PPT 专用的 ActiveX、COM、VSTO、或微软新版 JS Office 驻留插件盘。任何方案更改只支持在平台通过标准文书覆写上载。
- **不得加入完整企业知识库**：一律不建立多级复杂的企业非结构化向量数据库全局大 RAG 检索模型，只针对当前招标文件项目开展特定的提炼與智能一致性核验自测。
- **不得让 AI 自动替代专业判断**：AI 只生成建议，人工在前端页面未点击确认前，绝不擅自越过人工直达单一事实主数据库。敏感文件默认禁止 AI 读取，AI 读取前必须进行权限校验，禁止对敏感文件进行切片、向量化、摘要和问答。

## 二、开发过程规则

- **一次只开发一个迭代**：遵守顺序约束，在上一个迭代的功能完全联调、跑绿之前，禁止提前写后续迭代的代码支架（例如：在迭代一、二主数据和 AI 转换不稳前，不允许编写一致性自检逻辑）。
- **当前迭代未验收，不进入下一迭代**：每个迭代必须以测试报告绿通过，且向产品负责人演示、签字完备后，才支持进入下一轮研发分支合并。
- **每次代码变更必须说明修改文件**：在向平台提交成果或说明时，强制准确指名所做物理变迁的文件清单。
- **每次代码变更必须说明实现内容**：一律拒绝“优化代码”等虚无空洞提交说明，必须说明诸如“修复了建筑面积 PUT 保存时，未向 master_data_change_logs 对应追加变更历史的空指针 bug”。
- **每次代码变更必须说明测试结果**：每次开发新 API，必须附加该接口在 Postman 或集成环境中的真实回字测试。
- **每个迭代结束必须输出验收报告**：包含测试场景对照表、越权漏洞扫描绿。
- **每个迭代结束必须跑测试**：每次要确保 SQLite 的全套 API 本地脚本全部无损跑通。
- **发现规格或架构冲突的处理机制**：如果发现不同的产品规格、设计说明中带有相左的描述（例如高敏文档的 isSensitive 在不同说明里面分别写了 is_sensitive 与 isSensitive 字段不同）：
  1. **先记录，不要自行扩大解释或在代码里两个都写。**
  2. 应当向产品负责人进行 prompt 文字直接询问。
  3. 待获得白纸黑字的确切答复并在任务书中修正后再写代码。
- **需求池分离**：所有过程中的突发联想、新增“这个功能很酷”需求，一律记入 backlog，绝对不临时插入当前迭代分支中。

## 三、AI 开发规则

- **AI 服务必须可插拔**：后台禁止在业务类或控制层里直接 Hardcode 引用某一家专用 API 实体库（比如禁止直接 new Minimax() 甚至 hardcode api_key）。
- **通过统一 AI 服务层隔离 (AI Gateway Adapter)**：所有模块调用模型，必须经由底层专门设计的 `AIServiceGateway` 网关适配器进行中继转换。
- **当前可接 MiniMax-M2.7**：统一封契接口并接入 MiniMax-M2.7，模型参数设好温度防幻觉。
- **后续可任意替换**：为后续切本地小模型（如 LLaMA 或 DeepSeek）留置通用适配层规范。
- **AI 调用前必须检查文件权限和 AI 可读范围**：如果项目文书是高密或 `isSensitive == true` / `isAiAllowed == false`，AI 切片组件强制扔出 `AI_PERMISSION_DENIED_EXCLUSION` 并截断。
- **AI 输出必须包含来源文件、页码或段落**：抽取 prompt 的最终极约束是输出必须附带引证底页的 `sources` 行，以便右至左能够联动黄色高亮。
- **AI 结果必须人工确认才能入库**：绝对不在无人工点击 `POST /sync` 并签署 JWT confirmed 前，直接静默把提取项直接写进 `project_master_data` 表中。

## 四、权限和日志规则

- **后端路由强制鉴权 (API Security Level)**：不能只在前端通过 React 的置灰来防泄露，后端全部写接口（PUT / POST / DELETE）第一行强制调用 ACL 中间件验证：
  - 例如修改主数据接口，后端验证 role 必须属于营业员或负责人，否则斩断并直接吐 403。
- **文件下载必须校验权限**：对工程概算大卡下载 `/api/documents/download`，后端读取任务归属关系，非组内专职人员或非高管负责人均不放行物理下载。
- **AI 读取文件必须校验权限**：在 AI 解析前，鉴权模块物理核准本账号是否可以阅览此招标文件。
- **主数据修改必须记录日志**：每次覆写主数据，自动将前后老值、执行人、变天在 `master_data_change_logs` 中记录，任何人无权覆盖。
- **文件上传和下载审计日志**：文件版本采用 documents + document_versions 结构，历史版本永远不物理覆盖，对于文件的拖拽上传、新版本上传与版本历史保留，其 audit 专区必须留下详细的操作日志记录，如：`[LOG_FILE] xxx 上传了新版本 yyy_v2.docx`。
- **AI 调用消费 Token 审计**：AI 调用的 tokens 花销与响应消耗时长，安全记入 `ai_token_usage_logs` 表中长期审计。
- **审核意见状态流转与审核意见关闭**：审核意见状态流转与审核意见关闭过程必须在日志中全程记录。

## 五、完成定义 (Definition of Done)

每个迭代在宣告完成并发布演示前，必须无死角全数通过以下硬门槛（DOD 严格把关）：

1. **功能实现完成**：所有任务拆分书里明文规定的 P0 级页面、数据库表、Api、契约无一漏题地完工。
2. **权限校验完成**：多岗多账户登录，安全隔离，设计不碰报价，施工不瞧机密，未授权一律 403。
3. **日志记录完成**：去 SQLite `audit_logs` 和 control 后台查明，修改足迹和上传操作完备记录入库。
4. **测试通过**：本迭代所有的集成测试场景在集成环境 100% 绿。
5. **产品验收通过**：页面排字端庄高贵、符合 Space Grotesk/Inter 字体美学底蕴，无 Telemetry 无意义拼字乱码。
6. **业务演示完成**：在多角色演示环境中模拟整个大故事流程，完美展现本模块之商业价值级协同。
7. **已知遗留问题及 Backlog 记在册**。
8. **未进入下一迭代前不得继续扩展功能**：合并此分支并封闸。

## 六、系统设计核心开发规则

为确保开发落地的绝对工程质量与架构安全，整个 MVP 必须严格遵守以下三条正式开发规则：

- **规则一：主数据数值字段结构化存储**
  - 建筑面积、工期等字段不得只用带中文单位的字符串存储。
  - 必须拆分为数值和单位字段。
  - 示例：
    - `grossFloorAreaValue`：建筑面积数值 (REAL/NUMERIC)
    - `grossFloorAreaUnit`：建筑面积单位 (TEXT)，例如 `㎡`
    - `durationValue`：工期数值 (INTEGER)
    - `durationUnit`：工期单位 (TEXT)，例如 `日历天`
  - 后端负责存储这些分离字段，前端负责合并进行显示格式化，如展现为 `85,000㎡`、`400日历天`，以此保证数据的可计算性、可比对性，防范正则引擎匹配失败。

- **规则二：文件版本永不物理覆盖**
  - 废除任何对已有物理文件直接覆盖保存、或直接修改原文件元数据的物理动作。
  - 文件版本必须采用 `documents` 与 `document_versions` 的一对多双层物理结构。
  - 每次上传新版本时，必须生成全新的 `document_version` 行纪录与物理存储。
  - 历史版本不可被物理覆盖和删除：旧版本只允许标记 `isLatest = false`、`status = obsolete` 或 `status = archived`。
  - 最终版文件的确定，必须依赖数据库状态标记，不得通过覆盖物理文件来实现。

- **规则三：主数据变更影响范围只作用于活跃对象**
  - 主数据变更影响提醒与需复核状态标注，只作用于当前项目内的活跃任务、活跃资料和未关闭的审核意见。
  - 活跃状态包括：`TODO` (未开始), `START_ALLOWED` (可先启动), `WAITING_INPUT` (待输入资料), `DOING` (进行中), `PENDING_SELF_CHECK` (待自检), `PENDING_REVIEW` (待审核), `NEED_REWORK` (需修改), `PENDING_RE_REVIEW` (需复核), `HAS_RISK` (有风险)。
  - 以下非活跃状态不自动产生联动影响：`ACCEPTED` (已完成), `CANCELLED` (已取消), `ARCHIVED` (已归档), `OBSOLETE` (已作废)。
  - 已完成、已归档的对象默认不被打回需复核状态，但项目负责人具有手动将其标记为“需复核”的最高权限。
