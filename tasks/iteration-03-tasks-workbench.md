# BidWorks MVP 任务拆分 - 迭代 03：倒排计划、项目总控台与个人工作台 (Task Scheduling & Workbench)

本文件定义投标任务主轴以及岗位协同空间的开发。包含根据投标截止日期倒退工期节点、岗位分派、项目总控制台安全风险研判、以及个人编制人无干扰的个人工作台。

---

## 一、迭代目标

说明本迭代要解决什么问题，以及本迭代完成后系统应该具备什么能力：
- 变传统的“先排期后截止”为“以终为始后撤倒排计划（Backward Scheduling）”：项目负责人只需要输入项目主数据中投标截止日期，排程引擎能够自后向前逆向演算出各专业编制开始、截止节点，并带有防工期倒挂、不合理时隙截击卡死验证。
- 自动提取资料生成任务清单：基于主数据特征与 AI 在迭代二中抽离的“标书特殊资料清单要求”，自动初始化并生成对应的各岗位任务列表。
- 多角色隔离协作空间：
  - **总控台**：向项目负责人、高管呈放极简的全案汇总盘（谁未配、有何延、何处待复核）。
  - **个人工作台 (Workbench)**：作为设计、施工等编制专员的个人专属工作桌面，只保留其职责内的“我的编制文档、我的待办审核、主数据重变提醒”，不堆叠无关的大盘报表。

## 二、功能范围

1. **倒排计划自动引擎 (Backward Scheduling Engine)**：
   - 项目负责人录入或引入 `bidClosingDate` 投标截止日期后，系统自右向左自动铺设：合并总装技术方案（7天前开始，3天前截止）、编制设计方案（共需5天，10天前开始，5天前截止）、编制概算底卡（共需4天，11天前开始，7天前截止）。
2. **排程常识逻辑校验拦截 (Boundary Logic Validators)**：
   - 如果算出的阶段开始日期在当前落立项之前，或者倒退时限产生了重叠交叉、出现了负大数。
   - 系统安全截断重置并提醒警告，防止数据混乱存盘。
3. **特制资料物料任务包派生 (Tender Checklist Derivation)**：
   - 自动生成基础任务：生成设计组织设计、商务报价卡、施工大纲编制基础大卡片。
   - 招标特殊任务：如果在迭代二中 AI 提出了类似“要求附带特种设备租赁保证”特殊要求，此处在任务清单表中衍生增加专属任务项。
4. **负责人与结构化审核人指派 (Assignees & Reviewers Bindings)**：
   - 支持项目负责人进行卡片认领分派。指派某人（王五）作为 `设计大纲编制任务` 的 Assignee，以及指派领导（钱八）担任对应的专属审核人（Reviewer）。
5. **项目负责人基础总控台 (Control Panel Base)**：
   - 项目健康全局视窗：总览目前派员完整性，直观警告濒临超期临界点。
6. **编制员个人工作台 (Workbench Desktop)**：
   - 普通登录员工一页：显示我的编制列表、我的审核流待办、主数据重大变更报警（需复核红标提醒）。

## 五、前端任务

1. **一屏控御风险——项目总控台（Control Panel View）**：
   - 界面列出四项主体工件（设计方案、施工技术方案、工程概算底盘、商务装卷）。每个工段列出：专职责人、分指审核官、投标红线倒计时、以及任务自循环状态。
   - 对临期指标使用显眼红色字体作“濒危”通告，去掉不成熟的模拟测速计等视觉冗余。
2. **岗位及领导指派设定控制 (Assign Dialog)**：
   - 负责人点击时拉起小弹窗，快捷选择组内经过审核权限和对应专业的人。例如对于王五绑定 role 为 `DESIGN_ROLE`（专职设计），避免给设计人员配施工组织编制权限。
3. **普通员工专属个人工作台 (Personal Workbench)**：
   - 王五个人主站。只放三个标签区域：
     - `我的编制（In Progress / Rework）`：正在由我编写的稿页。
     - `我的待办审核`（如若王五是领导，则能在此看见其他人的待办审核清单）。
     - `主数据更新警告`：如面积从85000更正为90000了，在此高亮弹出一句话，标明“有面积变更、你的方案需要复核！”

## 六、后端任务

1. **倒排排程算法与时序校准守护 (Temporal Guard)**：
   - 逆序铺设天数逻辑。如算出 `mergeTaskDate` 开工在立项期之前：
     - 系统抛 400 警告：“计算出来的开始时间非法，不匹配，排期失败”。
2. **专岗位鉴权拦截拦截器 (Task-Level Authorization ACL)**：
   - 在执行 `assign` 派单、或是专员在工作台提交进度写接口时，后端强制加载在库指派记录：`if (task.assignee_id != currentUser.id && currentUser.role != 'BM') return 403`。严防王五在库恶意越权写钱八的工件。

## 八、数据表或实体变更

本迭代在 SQLite 内引入任务计划及关系映射表：

```sql
-- 投标任务大表
CREATE TABLE project_tasks (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    task_name TEXT NOT NULL, -- 标准名称：施工方案编制、工程概算底卡、设计大纲
    assignee_id TEXT, -- 指配工编制专员的 ID
    assignee_name TEXT,
    reviewer_id TEXT, -- 指定对应的专属审核人领导 ID
    reviewer_name TEXT,
    start_date TEXT, -- 倒排算出的开始时期 (YYYY-MM-DD)
    end_date TEXT, -- 倒排算出的截止日期 (YYYY-MM-DD)
    status TEXT NOT NULL DEFAULT 'TODO', -- TODO:未开始, DOING:进行中, PENDING_CHECK:待自检, PENDING_REVIEW:待审核, ACCEPTED:已定稿, NEED_REWORK:需复核状态
    task_source TEXT DEFAULT 'SYSTEM', -- SYSTEM:系统预置, TENDER_SPECIAL:AI从标书抽离的特殊文件要求
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- 任务前置及节点关系映射 (初版，为高密前后置做底座支架)
CREATE TABLE task_predecessors (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL,
    predecessor_task_id TEXT NOT NULL,
    FOREIGN KEY(task_id) REFERENCES project_tasks(id) ON DELETE CASCADE,
    FOREIGN KEY(predecessor_task_id) REFERENCES project_tasks(id) ON DELETE CASCADE
);
```

## 九、接口清单

### 1. 建立倒排排期计划 API
* **方法**：`POST`
* **路径**：`/api/projects/:id/tasks/schedule`
* **参数**：`{ "bidClosingDate": "2026-07-16" }`
* **返回**：`{ "success": true, "tasks": [ { "id": "t1", "taskName": "设计方案编制", "startDate": "2026-07-06", "endDate": "2026-07-11" } ] }`
* **权限**：仅项目负责人可发起排，其他人拒绝。
* **日志**：系统操作底案：`[LOG_AUDIT] 项目负责人李四发起逆推工期自动折返排程, 截止日期为: 2026-07-16`

### 2. 卡片派人及审核领导绑定 API
* **方法**：`PUT`
* **路径**：`/api/projects/:id/tasks/:taskId/assign`
* **参数**：`{ "assigneeId": "u5", "reviewerId": "u8" }`
* **返回**：`{ "success": true, "message": "指派完成" }`
* **权限**：项目负责人
* **日志**：`[LOG_AUDIT] 李四指派陈七担任[施工技术方案编制]编制人, 钱八担任专属审核人`

### 3. 获取登录专员纯化个人工作台 API
* **方法**：`GET`
* **路径**：`/api/users/workbench`
* **返回**：清洗后、无杂质的个人待办编制、待审卡、重变报警。
* **权限**：登录用户
* **日志**：只读，轻量不记录高管级 audit。

### 4. 获取项目负责人总控制台大分析 API
* **方法**：`GET`
* **路径**：`/api/projects/:id/control-panel`
* **返回**：项目内四模块、延误率、变更影响提醒与高风险计数全案汇总大盘结构。
* **权限**：营业员 (BD) / 负责人，禁止普通专员直接调用。
* **日志**：无特殊审计（常规读取）。

## 十、权限和日志要求

- **多端分权与数据隐形保护**：
  - 后端在 `GET /api/users/workbench` 与 `/api/projects/:id/control-panel` 层中严设数据过滤层。编制人员陈七只能查获到与其职责直接绑定的项目行，不可查看其他项目的商业报价秘密（钱八编制的工程概算）。
- **流程事务操作阻拦记录**：
  - 指派、负责人调期倒排等重要节点物理行为必须持久化审计日志归卷，保障发生问题时其变更原因可回溯。

## 十一、验收标准

1. **逆向折算倒排高准确无倒挂**：李四设置截止期：`2026-07-16`。触发排程，后台毫秒级退演，精确把概算编制起止划为 `2026-07-05 ─► 2026-07-09`，无时序负数倒错。
2. **专员工作桌面数据隔离**：王五（设计）账号登录打开系统，其桌面无任何施工技术方案的“待办”或 “项目总控台高风险提醒”，仅仅在自己的“我的指派”里展现了唯一一排 `【设计方案编制任务卡。状态：TODO，起止：2026-07-06至2026-07-11】`。
3. **任务包派生准确无残留**：在迭代二中 AI 在文本第 14 页判定招标方强制要求有“中德环境认证证书”。建项任务派发时，任务库自动多出生动的一条 `【准备特殊中德认证资料任务】` 被强制挂陈七，任务源精准归类为 `AI_SPECIAL`。

## 十二、测试场景

### 场景 1：倒排工期开始时期重叠越红卡死测试
- **测试前置条件**：账号项目负责人李四已成功登录，项目立项在 2026-05-20。
- **操作步骤**：
  1. 李四手动设定投标截止截至由于很紧急强设为 `2026-05-22`，并点击触发逆算倒排。
- **预期结果**：
  - 计算模块运算时，发现扣除 7 天后，其设计任务的开始时间将是 `2026-05-15`（在立项日以前，或在今天以前）。
  - 后端直接卡死熔断拒绝保存，往前端报：`"TEMPORAL_CONFLICT: 工期倒挂缓冲极其匮乏，开始日期非法！"` 界面警告并引导其手动压缩。

### 场景 2：基于设计专员与施工专员多角色数据隔离与越权阻断测试
- **测试前置条件**：王五（设计专员）、陈七（施工专员）、钱八（概算人员）均在库中。
- **操作步骤**：
  1. 陈七以其物理 Cookie 尝试发起 `GET /api/projects/p1/control-panel`。
  2. 陈七再次发请求查询在钱八名下的工程概算设计底卡 `t3`。
- **预期结果**：
  - 拦截器检测到其职位为施工，第一个大盘接口直接回绝 403 Forbidden。
  - 第二个查档也被系统直接过滤，返回：“该方案超出个人职责范围。您不具备读取权限”。

### 场景 3：招标文件 AI 提取特定材料清单到任务物理行产生测试
- **测试前置条件**：迭代二 AI 解析生成过一条带 sources 的招标附件备忘：“需提交项目章程授权”。
- **操作步骤**：
  1. 负责人李四点击一键指派激活。
- **预期结果**：
  - 检查 `project_tasks` 表，除了有经典的三大方案编制在库，准确增加了一行：名称：“需提交项目章程授权”，任务源标注为 `TENDER_SPECIAL`，依赖并附带着当时的引证页码 “4”。

## 三、风险和注意事项

* **时序与假日问题带来的工期压缩**：真实的排期通常不除周六、日可能会被编制人员反弹。
  * *处理原则*：MVP 阶段统一按标准的连日（日历天 Calendar Days）逆推运算，不引入法定清明、春节等中国法定节假日和穿插串盘多重判定，保持后端引擎轻量直接，防止排程陷入无限运算死循环。
* **人员角色变动**：
  - 编制人离职重绑：支持项目负责人在截止日期前在卡片下拉里任何时候对指配人重新解绑和重指派。
