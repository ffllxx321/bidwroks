# BidWorks MVP 任务拆分 - 迭代 04：文件版本、资料上传与提报前智能自检 (Versioning & Self-Check)

本文件定义文件管理与提报前组件智能自检引擎的系统方案。包含新版本上传、历史版本保留、历史隐藏折叠、以及在提交审批一秒前快速跑网比对并指出项目名偏颇、主面积数据误差、工历天不符和未脱敏黑名泄密残留等不合规行，并支持专员撰稿不处理原因并在日志强制留档。

---

## 一、迭代目标

说明本迭代要解决什么问题，以及本迭代完成后系统应该具备什么能力：
- 解决投标编制文本流中的“脏数据残留”与“重变不符常识”：编制人在其 Workbench 的指派任务（如施工方案编制）下，可以拖入并上传本地完成的 Docx 或 PDF 文件。
- 自动化版本升级（Automatic Versioning）：每次上传新版本都生成新的 document_version，老版本采用 documents + document_versions 结构，历史版本永不物理覆盖，旧版本只允许标记 `isLatest = false`、`status = obsolete` 或 `status = archived`。提供折叠手风琴面板可读浏览。
- **极致提报前自检 (Strict Live Pre-Check)**：在点击“提交审批”给领导看的一秒前，运行纯本地字符扫雷器：
  - 项目名称一致性（校验正文文字是否和项目主数据库里的项目名称吻合）。
  - 建筑面积一致性（校验正文文字里的多处面积，必须是项目主数据库里的建筑面积数）。
  - 工期限定天数一致性（校验正文里大标写的工期历时日是否吻合）。
  - 脱敏脱密硬拦截（检索正文里是否含有前次历史遗留项目的废弃敏感词字典“徐汇”、“张江”等）。
- 自检引证与手动反驳：自检出的问题列表里将清晰标绘具体位置，并支持专员撰拟抗诉原因强行忽略（限制：致命的脱敏黑名单词汇绝对拦截，不容忍任何忽略）。

## 二、功能范围

1. **多岗方案上传与任务挂接 (Upload & Tasks Binding)**：
   - 上载程序。自动关联在当前激活状态的项目，在提交文件时，专员手动下拉选取其所属的物理方案分类（例如“设计大纲任务”）。
2. **多态物理版本迭代 (Automatic Handshaking Versioning)**：
   - 系统接收到同类任务文件的复写后，自升 `v1 -> v2 -> v3`，并重新转换新名称；先前版本改为非 active 且只读废弃备忘态。
3. **一致性检查引擎 (Consistency Matcher Engine)**：
   - 后端切字将文档（Docx、PDF）在内存转为纯字符串，开展精准正则、模糊字及 Levenshtein 字符距离重合同度比对，发现数值差、不吻拼写时警告。
4. **涉密黑名单脱密检索 (Sensitive Decoupling Guard)**：
   - 提取正文中所有词流，比照敏感黑名字字典，一旦命中徐汇、大同等特等词，将其强行定为 `MUST_RESOLVE`（高危必改）。
5. **定制标书清单特殊响应检查初版 (Tender Checklist Matcher)**：
   - 检测正文中是否存在含有“中德环境认证”等迭代二 AI 自动在标书里抓出的物资大项字干。如果没有，自检红字提示：“标书中含有此资格限制项，您的技术方案内似乎还未进行妥善应答与编写，请核实！”
6. **自检问题来源显示与手动豁免记本**：
   - 显示报错，并对部分警告（如工期与主数据有 10 天误差，专员可在文本框录入“由于设计穿插所以在此压缩了”这排免死辩由）提供忽略开关。忽略按钮带有最低字数限，全量向 audit 写出。

## 三、不包含内容

1. **在网页端进行完整、复杂的 Office docx 在线协同修改、带修订轨的光标展示**（此由用户在本地 office 工具中完成修改）。
2. **Word 全能大模型全文本重构一键改错套写**（防止 AI 误判生成带有伪逻辑的严重工程废标漏洞）。
3. **与任何外部共享网盘、NAS 多物理存储目录实时同步**。
4. **高风险涉密徐汇地名自动豁免忽略**（最高拦截机制，绝不开发例外放行通道）。

## 五、前端任务

1. **版本更迭手风琴面板 (Version Archiver Accordion)**：
   - 处于任务栏最右测，仅突出显示 `[当前正在活跃的大方案 v3]`（最新 CURRENT 状态）。
   - 下侧放有一个“展开历史版本”灰色小标。用户点击拉开，可看到历史上所有 obsoleted 老版本文件的重命名情况、上传责任人和上传时刻，并支持只读下载。
2. **极简提交前自检控制台 (Pre-check Live Dashboard)**：
   - 陈七点击“提交送审”按钮，系统在幕布拉起一个精巧的扫描透视窗。
   - 展现三大严审栏目：
     - **大数字一致性一致性自测（数据面）**：如果项目主数据为 91000㎡，而 Word 里被粗心的人大写成了 85000㎡，高亮红体警告：“检测到建筑面积与主数据库冲突（主是91000㎡，你的正文写了85000㎡，位于第 4 页），需更正！”
     - **泄密字扫描（安全面）**：红色最顶级警告 `"在方案第14页发现历史非本工程敏感词‘徐汇’，强制禁止提交！"`。
     - **特殊招标文件响应测试（招标文件面）**：警告 `"该标书第2页强制含有对特种租赁证书应答，检测到本草稿全文均无此类词，疑遗露响应！"`
3. **辩护原因录写对话框 (Override Debrief Modal)**：
   - 点击忽略警告（不包括敏感词），必须弹出多行输入框强迫编制员输入 5 字以上客观说明，方可解除置灰。

## 六、后端任务

1. **上传重命名命名控制策略 (Secure Rename Policy)**：
   - 强行修改上传文件，重新计算其哈希值并统一物理保存，确保老旧数据无法通过欺骗后端修改，只能升级 `version_number`。
2. **多核心过滤校验拦截卡点 (Deep Match Regex Checkers)**：
   - 面积检查正则：`[0-9,\.]{2,}\s*(?:平方米|m2|㎡)`，发现其与 `grossFloorArea` 数差绝对值不等于零时报错警告。
   - Levenshtein 极值检查：把项目名与 `projectName` 做拼写对比，拼写错误或拼音混入直接报错并标写高危。
3. **高危必改强制截断拦截器 (Must-Resolve Guard)**：
   - 在忽视 `ignore` 接口入口的最上方进行第一极验证：`if (issue.type == 'SENSITIVE_WORD') return 403`。高密徐汇、大同地名黑字任何人员无权放行。
4. **日志拦截防脱水：每一次由于忽略而产生的提报，其系统操作日志要记录在底库备份中。**

## 八、数据表或实体变更

本迭代新增文件多版本、以及一致性自检暂存库：

```sql
-- 文件多版本存储库
CREATE TABLE document_versions (
    id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL,
    version_number INTEGER NOT NULL, -- 1:代表v1, 2:代表v2
    file_path TEXT NOT NULL,
    file_name TEXT NOT NULL,
    is_latest INTEGER NOT NULL DEFAULT 1, -- 0:作废Obsolete, 1:CURRENT最新
    uploaded_by TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(document_id) REFERENCES project_documents(id) ON DELETE CASCADE
);

-- 自评一致性错误数据库记录
CREATE TABLE doc_self_check_results (
    id TEXT PRIMARY KEY,
    document_version_id TEXT NOT NULL,
    issue_type TEXT NOT NULL, -- AREA_MISMATCH:面积不合, DURATION_MISMATCH:工期冲突, SENSITIVE_WORD:遗留脱密敏感词, TENDER_UNRESPONDED:招标文件特例未响应, NAME_MISMATCH:名字错字
    severity TEXT NOT NULL, -- FATAL:致命(如敏感字), WARNING:中度警告(允许手写理由放行)
    issue_description TEXT NOT NULL,
    found_text TEXT NOT NULL,
    position_citation TEXT, -- 在文书里的物理引段
    is_ignored INTEGER NOT NULL DEFAULT 0, -- 0:未豁免, 1:已被人工用言写忽略
    ignore_reason TEXT,
    ignored_by TEXT,
    ignored_at TEXT,
    FOREIGN KEY(document_version_id) REFERENCES document_versions(id) ON DELETE CASCADE
);

-- 脱密项目敏感词词典库
CREATE TABLE sensitive_black_dictionary (
    id TEXT PRIMARY KEY,
    sensitive_word TEXT NOT NULL,
    replacement_hint TEXT
);
```

## 九、接口清单

### 1. 挂接方案且版本自动升级物理上传 API
* **方法**：`POST`
* **路径**：`/api/documents/upload-revision`
* **参数**：Multipart 带项目ID、指配的任务 taskId、以及最新 docx。
* **返回**：`{ "success": true, "docId": "doc-88", "versionId": "ver-3", "versionNumber": 3 }`
* **权限**：仅限于该任务指派人（陈七、王五等）或者项目负责人，其他闲杂专员拒绝。
* **日志**：上传新版本：`[LOG_VERSION] 用户陈七成功上传了新版本，历史版本保留且前期版本2已作废折叠归档`

### 2. 激发本地一键智能自检 API
* **方法**：`POST`
* **路径**：`/api/documents/:versionId/self-check`
* **返回**：诊断问题明细列表 JSON 包（包括面积错、徐汇残留、工段天数不符等）。
* **权限**：登录人。
* **日志**：`[LOG_SELFCHECK] 专员陈七对 doc-88 的第三版发动了一致性扫瞄自查, 捕获到 2 个警告， 1 个高危致命词汇`

### 3. 主观反驳理由客观忽略 Api
* **方法**：`POST`
* **路径**：`/api/documents/self-check/ignore`
* **参数**：`{ "resultId": "res-3932", "ignoreReason": "由于本案采用装配式，工期比项目主数据少 10 天属于可控战略" }`
* **返回**：`{ "success": true }`
* **权限**：仅对非 FATAL 类型的 issue 放行、强制要求 `ignoreReason` 有 5字以上。
* **日志**：**高敏审计**：`[LOG_SELFCHECK_EXCEPTION] 陈七对工期与主数据不符进行了手写豁免忽略！理由是：由于本案采用装配式，工期比项目主数据少10天属于可控战略`

## 十、权限 and 日志要求

- **最严格的脱敏字不放行边界**：
  - 后端接口里，若 issue 为 `SENSITIVE_WORD`（敏感徐汇、大同等），直接物理阻击，即便王五强送参数，依然抛 `403 ACTION_DENIED`，强制不可忽略。
- **反驳行为全程监控日志**：
  - 每一次人工刻意豁免面积、工期等差异，均在 `audit_logs` 备份，成为管理监督的核心底座防御，任何人无法静默撤下警报。

## 十一、验收标准

1. **自动升阶折废无漏洞**：陈七对同一施工方案提交两遍。在 SQLite `document_versions` 中，版本1的 `is_latest` 变为 0 (OBSOLETE)，版本2变为 1 (CURRENT) 并且文件在物理盘不留冗余杂乱重命无失。
2. **三维度大容侵全检测跑通**：王五投递的方案里拼错项目名称（叫“青普标段”，主数据是“上海青浦一标”）且含有废大词“徐汇”。执行自测：
   - 面积、项目名高亮弹出。
   - 徐汇红字爆发，且忽略小锁置灰强制不可忽略。
3. **忽略抗辩必填留档验证**：对于面积相差 2㎡ 误差，王五在忽略输入框什么都不填，点击确定报错；填入 6 字“允许此微小面积浮动合理”后，解除灰锁，一键可以提交。

## 十二、测试场景

### 场景 1：致命涉密地名敏感词绝对禁止忽略强截击测试
- **测试前置条件**：系统内置敏感词库已含 “徐汇”。段落中含有“本期方案参照徐汇二期工程”。
- **操作步骤**：
  1. 陈七发起自评自测。界面成功红字检出问题 issue-889：在 23 页含敏感残留徐汇。
  2. 陈七不肯改文书，在 Postman 强行发 `POST /api/documents/self-check/ignore` 指向该 ID。
- **预期结果**：
  - 后端过滤器极速反应：`issue_type == 'SENSITIVE_WORD'` 强制高危，直接打回：`{ "error": "ABSOLUTE_SENSITIVE_FATAL: 敏感涉密字强制禁止人为忽略，必须更正文档原文重新提交！" }`。

### 场景 2：数据一致性自测允许录白反驳忽略放行测试
- **测试前置条件**：项目主数据里总工期设定为 400 天。陈七的施工大纲 Word 写成了 `380 天`。
- **操作步骤**：
  1. 自评检出工期错：大数据库400，文书380。
  2. 陈七在忽略框录入：“考虑到大干穿插，决定压缩20天，合理”，并点忽略。
  3. 点击提交送审。
- **预期结果**：
  - 忽略顺利落底，此工期报警变为置灰，陈七被限制的“提交至钱八领导审核”大控制点顺利解锁，陈七完好完成了本次送审。

### 场景 3：非前线人员恶意覆盘及串乱权限测试
- **测试前置条件**：施工陈七本人的编制任务是 t5。设计王五登录，尝试向陈七的任务里提交新版。
- **操作步骤**：
  1. 王五请求 `POST /api/documents/upload-revision` 传入陈七的任务ID $t5$。
- **预期结果**：
  - 后端直接被防越拦截器阻击，返回 403 Forbidden，提示：“抱歉，您不属于本工件受指定编制人，无法向此任务挂载新文档！”

## 十三、风险和注意事项

* **大数字抽取模糊漏报错过率**：有些文书的面积写在类似“基底面积”等多极小项下，正则极易张冠李戴报警。
  * *处理原则*：系统对由于排版差异引发的面积、工期不符，以“黄色 WARNING”警告，允许他们使用手写客观理由的辩白单进行豁免，防止因为算法误报错卡死正常的赶标进度。
* **物理文件溢出大漏洞**：
  - 开发团队为了支持历史版本保留，会在系统内存大量文件，极易塞满物理托管硬盘。
  - *处理原则*：对 obsolete 大文件限制其留存最多 3 个版本历史存根，多余的在 SQLite 中把 status 打为 OBSOLETE_PURGED 并物理自动抹干，减容系统损耗耗。
