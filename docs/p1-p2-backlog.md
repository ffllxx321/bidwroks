# BidWorks 长期规划与 P1/P2 中长期技术储备底册 (P1/P2 Feature Backlog & i18n roadmap)

本技术文档对 BidWorks MVP 终验成功后，因安全合规、工程去戏谑化、多维度防爆设计以及 AI 安全边界需要而剥离、降级至 P1/P2 的需求项进行了系统的整理。形成了下阶段持续迭代的功能 backlog 与技术攻坚路径。

---

## 一、 优先级与规划评定规则 (Prioritization Strategy)

- **P1 (高优先级储备，试点后首推)：** 对一线的协作体验有重大增量、能够有效缓解网络并发和在一定程度上提升数据可读性的特性。
- **P2 (中低优先级，中长期规划)：** 涉及超纲算法（如复杂顺延计算、排班图等）或是辅助性质的大模型生成式长文本编写、报表大屏。

---

## 二、 长期迭代功能列表 (Feature Backlog Registry)

### 1. 【P1 级储备】多语言实时切换支持 (i18n & Global Language Toggle)
- **需求定位：** 引入全生命周期多语言框架，支持海外投标团队一键将整个系统从中文转换为英文或其目标地区语言。
- **功能设计：**
  - 使用 `react-i18next` 并在前端持久化区域存储语种选择。
  - 机器内部的状态标志（如 `pending_review`, `completed`）无需变动，通过底层翻译配属资源文件。
  - 数据模型层：支持双语录入项目主物理数据。
- **潜在价值：** 扫清该系统出海或在跨境合伙项目部的协同盲点。
- **潜在风险：** 多语翻译对特定复杂技术指标（如“反弯系数”、“超深阻尼”）可能带来歧义偏差，必须配备高标人工纠错词典。

### 2. 【P1 级储备】PostgreSQL 多活高并发配置及分布式多版本持久化文件系统 (Persistent Distributed FS)
- **需求定位：** 替换本地 SQLite 单机锁缺陷，彻底释放服务端面对百人深化团队极速协同、狂写主数据库时的性能。
- **功能设计：**
  - 底层基于 Node-pg / Prisma / TypeORM 兼容框架。
  - 在大并发改写主数据并触发 cascade 重算需复核状态时，加注行锁、死锁重试缓冲机制。
  - 本地 `uploads/` 分区剥离，对接专属分布式对象存储。
- **潜在价值：** 实现系统从“单点单项目级 MVP 桌面工具”向“千亿级超大型工程局投标智脑中心”的物理级飞跃。
- **潜在风险：** 复杂的多节点会签与版本写争抢可能带来死锁或事务不一致。

### 3. 【P1 级储备】全量双向协同冲突判定 (Operational Transformation / WebSockets Real-time Collision Block)
- **需求定位：** 阻止多人在同一微秒内上传同一大纲文档的不同次代成果导致的“最终版（is_final）”版本覆盖悲剧。
- **功能设计：**
  - 引用 WebSockets 在前端对同一技术意见（review_comment）或同一分项任务加置“有同事正在编辑中”的实体感知标识。
- **潜在价值：** 有效防备异地协作拼抢出现的覆盖偏差。
- **潜在风险：** 网络波动或非正常离线会导致任务或会签意见被长时间挂起与逻辑死锁。

### 4. 【P2 级储备】基于图拓扑的多节点防错逆排期网络自动推算 (Advanced Topo Scheduling Engine)
- **需求定位：** 替换当前仅有 tree-block-based 简单前置卡死。
- **功能设计：**
  - 设计图论拓扑算法。在 PM 修改“最迟截标日期”后，联动在技术上通过前置任务的锁定系数，自动为陈七等所有人推算最优、最极限的编制开工时限界。
- **潜在价值：** 让项目负责人从人工手写倒排日的繁琐劳动中解脱。
- **潜在风险：** AI 擅自或算法过于激进地缩编成员的工期，会引发过度紧绷导致编制质量崩塌。

### 5. 【P2 级储备】历史投标文件交叉分析与报价/技术红板（Technical & Price Bid Analytics Workbench）
- **需求定位：** 基于 pgvector 向量数据库和企业私网历史成功中标标底，对进行中的工程主数据作极速历史拟合。
- **功能设计：**
  - 为销售和预算大师提供辅助“竞品分析看板”，支持通过 AI 进行一键偏差分析，起草合理优化建议和降费率点。
