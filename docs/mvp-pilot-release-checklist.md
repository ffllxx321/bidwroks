# BidWorks MVP 试点发布准备清单 (Pilot Release Preparation Checklist)

本准备清单旨在指导项目运维及实施团队在将 BidWorks 投标协同平台推向试点投标现场前，进行必要的环境、账号、环境指标、试点运作准则与语言审查。保证试点发布的平滑和数据的机密安全。

---

## 一、 环境配置要求 (Environment Setup)

在部署至试点项目部前，请运维人员对服务器环境实施逐项自核对：

- [ ] **Node.js 运行时：** 确保运行环境为 Node.js v18 以上或 v20LTS 版本。
- [ ] **物理主库配置：** 
  - 试点（pilot）和正式生产（production）环境一律强制使用 PostgreSQL 关系型数据库，并确保在环境变量 `.env` 中正确填写 `DATABASE_URL` 且配置了 `pgvector` 支持。
  - 严禁任何形式的 SQLite 静默回落降级机制，任何非 PostgreSQL 数据库连接（如缺少 `postgres://` 或 `postgresql://` 协议前缀）都将被系统拒绝并引发强制故障中断（fatal error）。
  - 本地微型开发环境（development 环境）方允许使用 SQLite 便捷自检验，以维持纯单机调试体验。
- [ ] **AI 网关接入（可选/自备）：** 
  - [ ] 确保在 `.env` 或 `server.ts` 容器拉启时，将 `AI_PROVIDER` 改为 `gemini` 或试点特派厂商名称。
  - [ ] 填写的 `GEMINI_API_KEY` 或目标 API 密钥确有余量，网络对宿主平台通畅。
  - [ ] 若在断网单机运行，一律在 `.env` 中指定 `AI_PROVIDER=mock`。
- [ ] **文件存储空间：** 
  - 为试点大附件准备不小于 50GB 的专属分区。
  - 项目上传和多态版本存储（`uploads/` 目录）具备写和不可覆盖属性。

## 二、 账号与角色权限配置表 (User Accounts & RBAC Setup)

试点期间，项目部建议配置以下 5 类初始实体账号：

| 初始登录名 | 试点拟设人员 | 委指角色 | 允许的核心职责 | 严禁的操作越线 (拦截) |
|:---|:---|:---|:---|:---|
| `user-pm-01` | 试点投标负责人 (如李四) | **ProjectManager** | 创立项目、设定工期与大组指标、分配任务、强制审查重设。 | 绝无，属全站项目级别最高安全权限官。 |
| `user-sales-01` | 市场经营科 (如张三) | **Sales** | 上传初始招标文件、AI 提炼对准、建立通用表格要求。 | 严禁查阅或更改技术与深化设计交付件、严禁闭合审核意见。 |
| `user-eng-01` | 施工图和设计深化 (如陈七) | **Construction** | 承领防震/机电等深化任务、上传新版、填写忽略自检错误说明、回复意见。 | 严禁更改任务计划日期（is_date_locked）、无权越界更改他人任务与意见状态。 |
| `user-rev-01` | 公司技术部专家 (如钱八) | **Reviewer** | 提出技术指标不合规指摘、给意见盖章归档、执行 close 与 reopen。 | 严禁越权替代编制上传技术大纲底稿，禁止更改主指标。 |
| `user-view-01` | 市场监督组/特邀专家员 | **Viewer** | 全能只读，查看本投标看板数据推进进度（只提供 GET 控制器）。 | 严禁任何 POST/PUT/DELETE 写库交互（一律会被 RBAC 403 阻断）。 |

## 四、 试点运作准则与守则 (Operational Best Practices)

- [ ] **“人类终审人”原则：** AI 提取的主指标（如总建筑面积、截标日期），**严禁在无人类专家核实下点击写入数据库**。每次 AI 识别必须由 Sales 或 PM 多重核对、修改并点击“人造确认”并留存变更分析。
- [ ] **“防覆盖”原则：** 所有人补充成果报告时，**严禁使用直接覆盖上传的方式抹杀历史版本**。必须通过“上传递增版本”（即 v1, v2, v3 机制）生成物理上的新记录。
- [ ] **“豁免说明”原则：** 如果存在一致性预检偏差（如老项目泄密词提示“徐汇”），**严禁直接放行**。必须在下方说明理由（需包含工程合理关联说明，且长度不小于 5 个字），将其流转为“已忽略（ignored）”后，方可由 Reviewer 关闭意见和核准归档。
- [ ] **“意见不合规”原则：** 任何人对深化资料提出修改意见后，修改动作和回复必须是 **多版本交付物和技术回复说明的多模态绑定**，让后期结算和二次核查在 review_status_logs 中有据可依。

## 五、 界面语言一致性专项检查 (UI Language & Consistency Audit)

在向试点人员分发终端前，实施以下专项界面文案排查：

- [ ] **前端无任何残留中英合流：** 检查系统左侧边Tab，顶部标题，以及每个按钮的文案。严禁用 “SelfCheck 预检” 类似形式，改为 “极速合规自检”。
- [ ] **状态与标签映射（Mapping）正常起效：**
  - [ ] 任务状态显示为：“未开始”、“进行中”、“审核中”、“已完成”，数据库内的 `not_started`, `in_progress`, `pending_review`, `completed` 不得溢出至前台。
  - [ ] 严重度评定正确转换展示为：“高”、“中”、“低” (取代 database 的 `high`, `medium`, `low`)。
  - [ ] 意见状态显示为：“开启中”、“已回复”、“已关闭” (取代 database 的 `open`, `replied`, `closed`)。
- [ ] **AI 起草警示通告完备：**
  - [ ] 在提取页面加注明显横幅：“本列表各项参数属于 AI 辅助提炼生成，仅作为起草建议，不可替代专家专业技术判定。必须人工确认放行。”
- [ ] **系统各角落不带有 Larp 戏谑词汇：**
  - [ ] 无 “Red Alert 红色警戒” 词汇，已校改升级为 “**逾期预警任务**”；
  - [ ] 无 “Control Room 总控室” 或 “Shield 防火墙” 等游戏或网安夸大文案，已转换为 “**项目驾驶舱**”、“**合规防御网**” 或 “**安全检验管道**” 等规范专业术语。

## 六、 试点起航前最后一公里确认 (Pre-Flight Checks)

在启动试点宣贯会前，主干工程师请确保：

- [ ] `npm run build` 无任何故障退出，静态资源能提供完美路由静态代理服务。
- [ ] PostgreSQL 试点主库通过一轮 `initialized` 实操且基础角色、安全词条等默认关系完全录入成功。
- [ ] 主控台在面对高密集更新主数据、重审被触发打标 `requiresReview` 时，能正常在系统 notifications 表内对钱八等目标角色发起精准代办卡包通知。

## 七、 试点成功退出与推广标准 (Exit & Success Criteria)

试点运行时间建议为：**4 倒排周 (约1个月)**。若满足以下三项标准，则认为本次 MVP 试点宣告成功并具备大规模推广建议：

1. **业务流全闭环：** 在试点项目部中，确实依托本系统成功流转了至少 **1 个完整投标项目、3 组特殊技术方案**；且全过程所有状态、历史纠葛、审批意见、回复细节、人工忽略皆被 100% 安全刻写进底册。
2. **零越界违规：** 审计日志中显示 0 例“非授权人强行关闭意见”或“不写理由无条件旁路”的违规漏洞情况。
3. **一线满意率：** 通过问卷反馈，项目经理与一线深化编制人员对“极速一致性纠错自检”和“主数据变更一键红线需复核”两个工具的好评率均不低于 **80%**。
