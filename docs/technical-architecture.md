# BidWorks 投标一体化生产平台 MVP 系统架构设计书 (Technical Architecture)

本架构设计书定义 BidWorks MVP 的核心模块划分、核心业务数据流、AI 服务隔离抽象层以及内网备份与部署策略。

---

## 1. 系统模块划分 (Module Division)

BidWorks MVP 平台由多个微模块及底层服务组成，以此保证复杂投标流的清晰拆解：

```text
    ┌─────────────────────────────────────────────────────────────┐
    │                       前端 Client (SPA)                     │
    ├──────────────────────────────┬──────────────────────────────┤
    │  个 人 工 作 台 (Workbench)  │  项 目 总 控 台 (Control)    │
    │  主数据面板  │  解析对照区  │ 任务树配置 │ 文件自检域  │
    └──────────────────────────────┴──────────────────────────────┘
                                   │
                                   ▼ HTTP (API Endpoint)
    ┌─────────────────────────────────────────────────────────────┐
    │                      后端 API Service (Node)                │
    ├─────────────────────────────────────────────────────────────┤
    │  User / Auth Service   ◄──►   Project & Membership Service  │
    │  Master Data Manager   ◄──►   Tender Parse Service          │
    │  Task Schedule Engine  ◄──►   Document / Version Controller │
    │  PreCheck (自检) Engine ◄──►   Review Thread Controller      │
    │  Impact Analyzer       ◄──►   Audit Logging Manager         │
    ├─────────────────────────────────────────────────────────────┤
    │                 统 一 AI 服 务 接 口 层 (AI Gateway)          │
    └─────────────────────────────────────────────────────────────┘
          │                                              │
          ▼ API SSL                                      ▼ Local Disk
    ┌─────────────────────────┐                    ┌─────────────┐
    │  MiniMax API / 本地LLM  │                    │ SQLite DB / │
    │    (进行切片与语义提取)  │                    │ 本地文件托管│
    └─────────────────────────┘                    └─────────────┘
```

### 1.1 用户与权限模块 (Auth & Permission Module)
*   负责平台端的基础登录、Session/JWT 令牌分发。
*   基于项目、角色、文件资产的三重访问控制引擎（ACL Interceptor），判断某客户端请求是否有权下载或查询特定任务的机密文件。

### 1.2 项目模块 (Project Module)
*   管理投标项目周期生命阶段（立项中、投标中、已提交、已归档）。
*   负责维护项目成员配置列表与分工（指派编制人、审核人与汇总人）。

### 1.3 项目主数据模块 (MasterData Module)
*   维护本项目单一事实库：包含建筑面积、总限期、质量与文明目标等核心字段。
*   处理字段的更新事务，将每次人工对主数据的篡改存入 `MasterDataChangeLog`，并标记字段来源及最新确认校验状态。

### 1.4 招标解析模块 (TenderParser Module)
*   承接 PDF/Word 源文件的流式解析与临时文本切片段（Chunks）建立。
*   配合统一 AI 服务，提取关键时间、数值参数及风险条款，输出可人工修改的“解析确认草案”，确认一致后方可对主数据库执行批量写入。

### 1.5 任务倒排模块 (Task Scheduling Engine)
*   自动计划引擎。一旦判定投标截止日期，系统基于内置的总承包工程标及商务编写基础倒推期限间隔（配置各工序历时天数），自右至左自动计算并铺设任务截止目标，且具有防日期逻辑倒置的拦截性逻辑校验。

### 1.6 个人工作台模块 (Workbench Module)
*   数据隔离与专属协作空间。面向一线专员，依据成员角色索引本人的“待办任务卡”、“今日待审核”以及“核心指标变更提醒”，减少查看非本岗多余信息的开销。

### 1.7 文件与版本模块 (Doc & Version Module)
*   对上传的技术跟商务资料进行后台磁盘高密关联、自动命名（规范为：`项目任务名_版本号_脱敏字样`）。
*   支持自动升级版本。当同一任务上载新版时，将历史实体打上“已作废（Obsolete）”状态并折叠，在页面提供一键历史比照。

### 1.8 提交前自检模块 (Smart Pre-Check Engine)
*   本地文件与主数据的异步文本匹配核对引擎。
*   主要对上载至暂存区的文件，跑三个基本自检校验：
    1.  **项目名称一致性**：正文文字对比主数据库项目代称。
    2.  **建筑面积一致性**：方案中书写的全部平米数数值与主数据库进行自动对标（如误差大于 1㎡ 触发标记）。
    3.  **总工期一致性**：校验正文工期数字。
    4.  **历史非本项目废地名筛查**：自动按内置敏感多项目黑名单库比照，找出“残留徐汇地名”等泄密隐患。

### 1.9 审核意见模块 (Review & Collaboration Module)
*   结构化审核系统。审核人在线划定文件的疑点行，建立待处理、已修改、已关闭的状态意见行，不满足所有意见全部“关闭（Resolved）”状态，系统拦截终稿标记行为。

### 1.10 信息变更影响模块 (Impact Cascading Module)
*   核心字段发生更新（根据中高风险等级分类）时，不强制静默改写各专业编制中的方案正文，而是自动生成 **变更影响评估卡**。
*   使相关联的其他专业资料标记为“需重新复核”状态，且向受影响的岗位负责人推送复核提醒，直到相关岗位的负责人核对并确认完成复核后，该待复核状态方能消退。

### 1.11 日志审计模块 (Audit Log Module)
*   处理系统运维审计：记录文件上传/下载人员客户端详情、IP来源、AI API 每笔 Token 耗尽记录、敏感资产调阅拦截。

## 2. 核心数据流设计 (Core Data Flows)

### 2.1 招标书上传到主数据确认数据流
1.  [营业] 上传招标文件 (PDF/docx)。
2.  `TenderParser` 执行底层文本抽段，建立带坐标的物理段落索引。
3.  系统过滤由于 `isSensitive` 的敏感策略。非敏感前提下，文本切片提交至统一 AI 服务层。
4.  AI 结构化抓取项目属性、大日期并打上 `【物理源文件第X页第Y段】` 引证页码。
5.  系统返回一列“待确立的主数据建议卡”，在前端通过双联视觉渲染，一侧可阅引证原文原文，一侧允许营业员编辑更正。
6.  [营业/项目负责人] 校合修改，点击入库，主数据库正式入表。

### 2.2 主数据变更到需复核状态与变更影响提醒数据流
1.  [营业/答疑补遗] 修改主数据库中的“建筑面积（grossFloorArea）”从 85,000 变更为 90,000。
2.  `ImpactAnalyzer` 模块识别到此字段属于“中风险等级变更”，检索底层依赖图，判定其对 [设计] 的设计大纲、[概算] 的预算、[施工技术] 的临设和施工大纲、[汇总] 产生关联影响。
3.  系统将上述所涉岗位的在库未提交或已提交资料对应的任务状态变更为“需复核状态（Pending Review）”。
4.  向设计、施工和概算等岗位负责人的 Workbench 推送变更影响评估卡（注明原值、新值、变更发起者、修改时间及一键修改指引链接）。
5.  在相关岗位负责人进行手动确认之前，总控台持续展示待复核待办，且在流程上阻断对其进行标记为“最终版”的操作。

### 2.3 文件上传到提交前自检数据流
1.  [施工技术] 自行拖入 `施组编写模板.docx`。
2.  文件落地受控托管路径，状态自动置为 `Pending Self-Check (待自检)`。
3.  用户点击“开始提交前自检”，系统调用本地自检引擎，对文本内容展开校验。
4.  利用精确正则和文本比对算法核对项目名称、建筑面积（校验 90,000㎡ 是否正确吻合）、总工期（校验 400 天是否一致）以及历史项目特有实体敏感词过滤。
5.  输出“一致性自测合规报告单”：
    *   指标冲突（例如第4页带有“85,000㎡”描述）：标识冲突具体位置与数据偏差。
    *   未脱敏词汇残留（例如第24页带有“徐汇二期”等敏感地名词汇）：给出错误告警并提供原文定位。
6.  用户在本地修改 Word 文件并覆盖更新，上传为 v2 版本，至此再次自检全部通过，系统解除锁定，允许送审。

### 2.4 提交审核到审核意见关闭数据流
1.  [施工技术] 正式提交送编，文件状态轮换为 `Waiting Review (待审核)`.
2.  [审核领导] 进入待评后台，系统在其工作台拉起。打开界面时，钱八在左侧可见原自检回报与忽略标注，右侧可在线划线选段发表具体结构化纠备意见，指派陈七，并打上 `意见状态: Active`。
3.  [施工技术] 看到待修复挂单。由于系统强切，其在本地修改并在系统中再次覆盖更新（系统重归置为 v3 版本），撰写已修改答卷。
4.  [审核领导] 重刷新，对比上版本差异，若钱八确认合格，点击“意见关闭（Resolved）”。
5.  当所有意见状态均重归 `Resolved` 后，该文案满足基础合规度，锁解并允许授权人标记最终版。

### 2.5 AI 读取文件时的权限判断数据流
1.  AI 进程需要基于 RAG 或文档摘要任务，向底层数据库请求读取文档 `Doc_14_A.xlsx`。
2.  `SecurityInterceptor` 自动执行两步验证：
    *   第一步：验证该 AI 任务的主调起操作用户是否有下载/查看 `Doc_14_A` 的权限；
    *   第二步：检验由于字段 `isSensitive == true` 造成的敏感屏障是否开启。
3.  若该文件敏感，而未被负责人强制配符“开启 AI 本案摘要功能”，拦截。
4.  返回 `AI_PERMISSION_DENIED`，该 AI 任务不会对该文件进行切块、向量化或总结。

### 2.6 敏感文件被 AI 拒绝读取的流程
1.  [概算负责人] 提交 `青浦基地钢构商务谈判降价底表.xlsx`，系统判定该文档隶属“敏感级别”任务，存盘入库自动置 `isSensitive = true` 且 `isAiAllowed = false`。
2.  任何系统层大盘 RAG 索引脚本（批量爬取提取项目多文件归集的向量脚本）或 AI 摘要功能启动，系统向文档列表发出 RAG 文本切片向量化服务。
3.  权限引擎检测到此文档涉密且 AI 未被许可开启，安全拦截。
4.  该xlsx的任何内容不流向 AI 数据切切片任务，其物理盘内文本数据物理屏蔽，不向量化，不分段，不在外部模型入参暴露，彻底切断外泄隐患。

## 3. 统一 AI 服务隔离抽象设计 (AI Service Gateway)

为了确保平台在未来的高可移植性，杜绝业务模块与第三方模型间的深度耦合。

```text
  【业务调用层】(例如 TenderParser, Validator, Summary)
         │  (不直接实例化具体 SDK 接口)
         ▼ (通过抽象参数发起：DocumentChunk, ModelAttributes)
  【AI 统一服务代理网关 - AIServiceGateway】
         │
         ├─► [策略规则一：检查文件 AI 可读范围与 isSensitive == false ] (安全门卫)
         │
         ├─► [策略规则二：将请求分发至注册的模型适配器组件层 (Adapter)]
         │
         ├───► [适配器 A: MiniMaxAdapter] ──► (当前云端调用 MiniMax-M2.7)
         │
         └───► [适配器 B: LocalLLMAdapter] ──► (未来内网部署时私有模型无痛替换)
```

### 3.1 核心契约原则：
1.  **统一契约层**：所有的业务场景（分析、自检、提取）均通过通用的统一 AI Gateway 进行请求分层，参数需为标准 JSON 体，系统拦截器首先执行权限检验（确保调用者非涉密隔离对象）。
2.  **来源定位强制性约束**：Gateway 的统一 API Response 数据格式定义中，强制要求必须带有页码页码引证数组字段 `sourceCitations: Array<{pageNo: number, sentenceText: string}>`，从基础代码结构上强推溯源。
3.  **零硬编码替换能力**：业务代码中绝无 `const client = new MiniMaxSDK()` 此类代码。替换模型仅仅是在 AI 适配器模块重配，达成业务零感知升级。

## 4. 部署与数据备份原则 (Deployment Guidelines)

### 4.1 环境分层建议
*   **开发环境 (Development)**：本地单 Docker 部署，SQLite 数据库挂载本地以方便快速清除或重建。
*   **测试环境 (Test)**：部署于内网独立的 VM/轻量云上，使用脱敏后的“青浦制造基地”真实招标书及施工组织方案副本，针对权限及变更影响工作流进行完整多角色仿真测试。
*   **试点生产环境 (Pilot)**：部署于建筑企业内网上网环境，开启精精细的系统安全 RBAC，挂接备份数据库与 NAS 文件系统。

### 4.2 本地私有备份与安全防护
*   **数据双重备份策略**：
    *   **主 SQLite 库**：每天凌晨4时通过脚本进行快照复制备份（保留最近30天历史）。
    *   **文件系统**：结合企业内网 NAS 挂载路径，保证所有上传的工件不仅存留当前系统的持久磁盘，同时镜像保存在备份存储中。
    *   **灾备回滚**：当发生重大不可控故障需要环境恢复，支持一键将特定某一项目的一系列文档及库快照回滚到之前特定一天的状态。

## 5. 核心技术风险清单与消减对策 (Technical Risks)

| 风险对象物理描述 | 技术产生原因 | 架构级消减防御对策建议 |
| :--- | :--- | :--- |
| **PDF 扫描版 OCR 解析不准** | 多页纸质打印拍照偏转、字迹糊乱导致分段解析失效。 | **OCR 渐进回弹退路**：系统对于判定为不可提取文本图层的 PDF 直接阻断，引导用户采用“无标书手工速建”以解燃眉之急。 |
| **AI 提取发生“漏报”或“参数张冠李戴”**| 标书内容分散，AI 对同类条款交叉引用时，可能导致理解偏差。 | **人工强校验（Human-in-the-Loop）**：AI 的提取结果绝对不直接、静默更新主数据库，必须经由营业人工一键校核确认后入库。 |
| **变动联动影响带来频繁的复核干扰** | 若不加约束，任何低风险微小变更都会引发下游全员接收重新复核的通知，导致流程冗余。 | **重要程度评级逻辑隔离**：对主数据字段设定清晰的重要度划分，仅对大面积、大工期等中高风险更变触发“需复核状态提醒”，低风险轻微修正仅记录变更日志，不干扰各岗位正常生产。 |
| **内网部署时 AI 模型处理缓慢** | 内网物理机 CPU/显卡等推理能力逊于云端算力。 | **大解析异步化交付模式**：将招标文件文本切分与解析转换为异步 Worker 跑网模式，通过前台排队进度指示与状态轮询，防范阻塞界面与并发堆积。 |
