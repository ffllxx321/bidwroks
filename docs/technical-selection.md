# BidWorks 投标一体化生产平台 MVP 技术选型设计书 (Technical Selection)

基于建筑总承包企业对内网部署、数据涉密安全性、文档解析精度以及 AI 支持敏捷落地等多重诉求，技术选型需要兼顾 MVP 构建的敏捷度，以及未来演进到内网私有化部署的长期维护架构。

本设计书提供两套具有代表性的技术栈方案，对比二者利弊，并确定 BidWorks MVP 阶段的最终推荐路线。

## 1. 技术选型方案对比

### 方案 A：快速 MVP 优先 (轻量级/云原生托管/敏捷开发)
此方案旨在利用现代高集成度全栈技术及云端 API，快速跑通闭环工作流，具有极低的初始开发成本和极高研发流转效率。

*   **前端**：React 19 + TypeScript + Vite 6 + Tailwind CSS（组件层采用 Radix UI 与 Lucide 图标提供高密度精美桌面设计）。
*   **后端**：Node.js + Express / Fastify (TypeScript Native + tsx 调试热重载)。
*   **数据库**：SQLite 存储轻量级工程、任务与文档元数据，对单文件操作天然零配置。
*   **文件存储**：本地文件目录，或轻量级 S3 兼容对象存储 (如 MinIO 云托管服务 / 腾讯云 COS)。
*   **文档解析**：基于 `pdfjs-dist` 与本地解析引擎拉取文本型 PDF 逻辑段，Word 依赖 `mammoth` 抽取富文本结构。
*   **AI 服务接口**：云端统一 API（当前使用云端 MiniMax-M2.7 接口进行文本切分与实体捕获）。
*   **权限与日志**：轻量级应用内拦截器 + Winston 结构化输出到本地特定安全审计日志文件。
*   **部署方式**：单容器 (Docker / Cloud Run) 整体交付，支持快速构建热启动预览。

---

### 方案 B：企业内网长期维护优先 (解耦化/高并发/自主可控)
此方案立足于建筑企业局域网或公司私有云物理机环境，完全去云化，软件全组件均为企业级私有可维护版。

*   **前端**：React 19 + TypeScript + Vite 6（前后端物理分离）。
*   **后端**：Java (Spring Boot 3 + Spring Security) 或 Go (Gin + Gorm)，提供高度稳固的事务与并发操作保障。
*   **数据库**：PostgreSQL 或 SQL Server (建筑企业内网常见，支持大型数据一致性锁机制)。
*   **文件存储**：企业内网专用 NAS、SAN 物理机存储，由本地 Nginx 作为拦截代理校验。
*   **文档解析**：集中式解析集群 (Java Apache POI 进行 docx 深解，基于 Python PDFPlumber 进行定制规则提取与坐标检测)。
*   **AI 服务接口**：内网私有微调落地的大语言模型 (如 Llama-3-70B 或 Qwen-2.5-72B 本地 vLLM 推理集群服务)。
*   **权限与日志**：精细化基于 RBAC 和 Access Controller 的接口安全框架，主备多活级日志库持久存储（ELK 归集）。
*   **部署方式**：K8s 集群多 pod 分离式部署，独立数据库及文件服务器备份。

## 2. 方案对比维度矩阵

| 评估维度 | 方案 A：快速 MVP 优先 | 方案 B：企业内网长期维护优先 | 差距度与演进建议 |
| :--- | :--- | :--- | :--- |
| **初期开发成本** | **极低** (1-2位全栈工程师12周可跑通) | **高** (需要前端、后端、运维等复合团队，周期16-24周) | MVP阶段首选方案 A，利用统一服务接口降低联调成本。 |
| **数据安全性与隔离性** | **中高** (API传输需网络代理，本地存储高密度) | **极高** (物理内网闭环，敏感文件不出网) | 在方案 A 中通过限制敏感文件向云端 AI 传输，达成一致。 |
| **系统响应时延** | **极低** (单实例运行内存消耗小，调用快速) | **中** (多微服务层级交互，开销大) | 方案 A 具备更流畅的业务交互速度。 |
| **本地私有化升级复杂度**| **低** (只需将第三方 API 替换为本地推理端) | **极低** (自始即为本地架构) | 方案 A 在架构上必须预留 AI 服务层抽象，以便随时无痛迁移。 |
| **系统稳定性与运维负担**| **极低** (无多组件集群维护风险) | **中** (需专职 IT 人员维护 PostgreSQL、AI 集群、NAS) | MVP 试点阶段不宜增加过多运维负担。 |

---

## 3. 最终推荐路线选择与理由 (Recommended Path)

首选 **“方案 A (快速 MVP 优先) 的精细化改进路线”**。

### 推荐理由：
1.  **最大化敏捷性**：作为一款 MVP 级产品，BidWorks 的最主要卡点在于验证“招标解析人工确库 + 倒排计划 + 提交前自检 + 审核消项闭环”主流程的使用粘性。选用 React 19 + Node.js (TypeScript) + SQLite 能够以最快速度完成原型代码验证，排除多套服务器间联调的阻力。
2.  **平滑可移植性（可插拔设计）**：通过精心设计的统一 AI 接口代理组件 (详见架构设计中的 AI 服务设计原则)，我们在方案 A 的代码设计中完全解耦了第三方大语言模型 (如当前接的 MiniMax-M2.7) 的业务嵌入。未来项目一旦转入正式企业级，只需轻度重写服务层连接私有 vLLM 本地模型接口，即可在不对业务、数据结构、一致性校验表做任何大改的前提下，实现一键内网替换升级。
3.  **零污染的敏捷数据安全机制**：即使在 MVP 的云托管预览状态下，亦能在底层机制上强行阻断敏感数据上载到云端 AI 接口（见核心数据流设计四），从而让方案 A 的安全水位完美具备和方案 B 媲美的防泄漏隔离能力。
