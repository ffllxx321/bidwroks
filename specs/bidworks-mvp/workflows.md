# BidWorks 投标一体化生产平台 MVP 业务工作流与状态流转 (Workflows)

本规格定义 MVP 核心业务的工作流转序列及变更级联通知响应逻辑。

## 1. 投标平台端到端闭环主工作流 (End-to-End Workflow)

一个标准的投标项目从新建到提交，需遵循以下严密的标准业务主路径：

```text
步骤一：建项发起 ──► [营业] 决定是【上传标书一键建项】还是【无标书手工速创项目】。
                                   │
                                   ▼
步骤二：标书解析 ──► 系统驱动 AI 针对上传的标书大本执行文本解析与关键字段抽取。
                                   │
                                   ▼
步骤三：主数据确认 ──► [营业/项目负责人] 审查 AI 提取草案，进行修正与【点击确认入库】。
                                   │
                                   ▼
步骤四：倒排与委派 ──► 计划倒排引擎重算任务各节点，[项目负责人] 派发对应任务责任编制、审核及汇总人。
                                   │
                                   ▼
步骤五：方案编制交付 ──► [各专业编制人] 领用工作，上传资料（自动命名和版本升级 v1 -> v2）。
                                   │
                                   ▼
步骤六：线上多维自检 ──► 编制人提交前点击【开始自检】，排查别字、面积与工期冲突、前人旧残留，生成修改单。
                                   │
                                   ▼
步骤七：结构化审核 ──► 自检通过后，正式打包送审，[审核领导] 在线批注结构化意见，多方状态在线轮变。
                                   │
                                   ▼
步骤八：修改与消件 ──► 编制人根据审核意见迭代文件并线上答复，[审核领导] 终审并亲自【点击关闭意见】。
                                   │
                                   ▼
步骤九：标记定稿 ──► 控制台验证所有前置意见皆已 Resolved，且最新指标自检通过，[汇总员/项目负责人] 标记“最终版”封案。
```

## 2. 操作工件状态全生命周期流转规则 (Job Lifecycle)

在全作业中，为了保障在前后并行，每个工件文档的状态均受强流转逻辑控制，防范越界与遗失：

```text
       上传文件
          │
          ▼
     ┌──────────┐            不通过 / 发现错别字旧地名
     │   草稿   ├────────◄──────────────────────────────┐
     └───┬──────┘                                       │
         │ 自检通过并通过                                │
         ▼                                              │
     ┌──────────┐                                       │
     │  已上传  │                                       │
     └───┬──────┘                                       │
         │ 送审审核                                     │
         ▼                                              │
     ┌──────────┐          审核领导打回并要求整改        │
     │  待审核  ├───────────────────────────────────────┤
     └───┬──────┘                                       │
         │ 审核领导开审 / 状态锁住                       │
         ▼                                              │
     ┌──────────┐                                       │
     │  审核中  │                                       │
     └───┬──────┘                                       │
         │ 审核不通过                                   │
         ├──────────────────────────────────────────────┘
         │ 审核意见全部关闭
         ▼
     ┌──────────┐
     │ 已通过审 │
     └───┬──────┘
         │ 资料汇总员核准并点击最终版
         ▼
     ┌──────────┐
     │  最终版  │ (至此本部分任务锁死，禁止任何覆盖修改，作成果合成)
     └──────────┘
```

**版本升级规则**：对处于 `Draft`, `Uploaded`, `NeedsRevision` 状态的任务，在同一上载通道再次上传文件时，系统自动拦截原文件置为 `Obsolete (已作废)` 移入历史卡片折叠，新文件覆盖继承父级 UUID 并自动派发 `VersionCode = parent.VersionCode + 1` 后重构名称，不让用户每次手工在文件名后缀写 “施组最后一版_最终版_修改版” 字段。

## 3. 主数据字段变更“需复核状态与变更影响提醒”工作流 (Impact Cascading Workflow)

一旦在投标截止日之前由营业、或由于收到业主的补充 Clarification 答疑，修改了项目主数据库的某个核心变量，系统不强制暴力改写各专业进行中的文案正文，二是通过以下**需复核状态与变更影响提醒工作流**：

```text
 变更触发：修改主数据数字 (如建筑面积：80,000㎡ ──► 85,000㎡)
                 │
                 ▼
 引擎识别字段变更重要等级 ──► 判断为：【中风险】级别影响
                 │
                 ▼
 生成影响单 (Impact Slip) ──► 将主数据变迁记录永久入库备考
                 │
                 ▼
 级联资料标记“待复核” ──► 把已经或进行中该项目关联的设计方案、施工方案、概算图表修改为“需复核”状态
                 │
                 ▼
 推送变更提醒 ──► 特定角色（如设计员、概算员、施工技术、资料汇总员）在工作台看到相关提醒一键跳转单
                 │
                 ▼
 岗位人工重新核算 ──► 各岗负责人审核并修改其 Word 正文，完成后再次点击“已全面复核确认”
                 │
                 ▼
 恢复正常状态 ──► 资料状态自动消退，恢复至正常定稿状态
```

## 4. 字段变更级别判定表

### 4.1 高风险 (High Impact) - 必须触发重新计算或项目负责人重新审核
*   **字段包含**：投标截止日期、总工期、招标工程范围、施工承包范围、合同付款条件、报价分界。
*   **受影响角色**：项目负责人、营业员、设计员、设计领导、概算员、报价高管、施工技术、资料汇总员。
*   **处置锁闭**：直接重算计划倒排表的倒退工期节点，在总控台产生高风险项预警（High Risk Alert）。

### 4.2 中风险 (Medium Impact) - 发往相关负责岗核算并强标记资料“需复核”
*   **字段包含**：建设地点、建筑类型、建筑面积、BIM规划要求、安全文明、质量目标。
*   **受影响角色**：设计员、概算员、施工技术、汇总员。

### 4.3 低风险 (Low Impact) - 仅生成变更列表日志，不单独提示阻断
*   **字段包含**：项目名称、业主名称、答疑截止日期、现场勘查。
