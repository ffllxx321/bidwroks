# BidWorks 投标一体化生产平台 MVP 场景验收标准 (Acceptance Criteria)

本规格定义 MVP 必须通过的六大核心业务验收场景。每个场景都有明确的流转步骤、预期行为和拦截阻断机制。

---

## 场景一：上传招标文件建项 (Upload-and-Extract)

*   **测试前置条件**：系统无“上海青浦日资制造企业基地项目”等项目，使用账号：[营业员-张三]，准备一个含有明确投标截止日为 2026-07-15 和建筑面积约 85,000㎡ 描述的文本型 PDF 招标文件。
*   **验收测试流程**：
    1.  [张三] 登录系统，在新建项目页面点击“上传招标文件建项”。
    2.  拖放该 PDF 招标文件，页面显示解析进度提示。
    3.  解析结束后，页面跳转至双列对照模式：
        *   **左侧**：展示 PDF 文稿预览。
        *   **右侧**：展示 AI 提取的字段建议卡片：`projectName = 上海青浦日资制造企业基地项目`, `grossFloorArea = 85000㎡`, `bidClosingDate = 2026-07-15`，点击这些卡片，左侧 PDF 预览能自动定位并高亮对应原文页码及段落（具有精确的源文件溯源引证）。
    4.  [张三] 发现业主名称字段为空，手动在右侧待确认编辑框中输入，并将投标截止日期修正为 2026-07-16，点击“确认入库”。
*   **预期成功标准**：
    *   项目被成功创建并初始化其主数据库，未发生空指针异常。
    *   在主数据详情页可查，字段来源分别记录“引自招标书第 4 页第 2 段”与“张三手动录入维护”。
    *   系统审计日志（AuditLog）中成功记录该笔 AI 调用引用可审计日志与主数据变更足迹日志。

## 场景二：无招标文件基础手工速创项目 (Manual-Setup)

*   **测试前置条件**：业主尚未正式发布招标文件，使用账号：[营业员-张三]。
*   **验收测试流程**：
    1.  [张三] 登录系统，点击“无标书手工速创”。
    2.  填写最基本的主数据必填项：`projectName`（临时代号：青浦BD）和 `clientName`（业主 XX公司），点击“确定创建”。
    3.  其余技术和商务指标字段（如建筑面积、付款方式）均自动填充为 `待确认` 状态。
    4.  项目首页主数据面板中，尚未确认的空白字段以虚线外框提示。
*   **预期成功标准**：
    *   成功为项目建立独立生产空间，且项目负责人随后能够正常为设计、施工指派任务。
    *   所有未确认的空白主数据自动在后续关联任务自检与总控台中标记为“未确认悬空项”。

## 场景三：核心字段建筑面积更迭“需复核状态与变更影响提醒”核验 (Cascading-Drift)

*   **测试前置条件**：项目已创建，主数据 `grossFloorArea = 85000㎡` 已被确认。使用账号：[项目负责人-李四]、[设计员-王五]、[概算员-赵六]、[施工技术-陈七]。其中设计员王五已提交设计说明大纲，概算员赵六正在计算中。
*   **验收测试流程**：
    1.  由于业主发布答疑书修正指标，[项目负责人-李四] 登录平台，在主数据详情页将建筑面积修改为 90,000㎡ 并保存。
    2.  系统后端启动中风险变更影响圈定工作流：
        *   自动将该项目下关联的设计方案说明书、工程概算底稿、施工组织方案等进行中或已完成的任务与对应文件状态变更为“需复核”状态。
        *   向主数据变更日志中录入原值（85,000）、新值（90,000）、修改人（李四）和修改时间。
    3.  [设计员-王五] 登录到个人工作台，其界面上方显著展示变更影响提醒：“主数据建筑面积发生变更（原 85,000 ㎡ -> 新 90,000 ㎡），相关资料已标记为‘需复核’，请核实调整”。
    4.  [王五] 点击一键跳转至方案对比界面，修改本地设计大纲文件，使技术指标对齐 90,000 ㎡，随后覆盖上传更新版本，并手动点击“已全面复核确认”按钮。
*   **预期成功标准**：
    *   只有在王五、赵六、陈七等受变更影响的各专业岗位负责人全部在平台内点击“已全面复核确认”后，该项目主控台关于此项面积变更的提醒标识才予消退。
    *   在此复核链条关闭前，汇总安全拦截器生效，阻断对该项目的任何文档标记“最终版”操作，并弹出拦截警告。

## 场景四：方案提交审核前智能一致性自检 (Zero-Fault)

*   **测试前置条件**：项目主数据确认面积为 90,000㎡，总工期要求为 400 天。使用账号：[施工技术-陈七]。陈七准备上传一个直接套用以往历史工程模板、文本中残留“总工期 380 天”及多处“徐汇”旧地名的施工组织大纲 Word 文档。
*   **验收测试流程**：
    1.  [陈七] 登录工作台，在分配给其的“施工组织编制”任务下，上传个人编制完的 Word 文件。
    2.  在点击送审前，陈七点击“开始提交前一致性自检”按钮。
    3.  系统后台异步自检引擎对文件文本进行校验，解析全文并识别到多处拼拼漏与不一致：
        *   `问题 1`：检测到正文中面积写着 “82,000㎡”，与项目主数据确认的 “90,000㎡” 不符。
        *   `问题 2`：检测到工期文本描述为 “380天”，与项目主数据确认的 “400天” 不一致。
        *   `问题 3`：检查出历史废套模板残留实体词 “徐汇”，涉嫌脱敏不全。
    4.  [陈七] 确认工期写为 380 天符合本项目专家的具体工期优化要求，因此在问题 2 右侧点击“忽略”，并手动填写主观理由说明：“本项目经技术论证，拟将 380 天提前交付作为技术亮点”。对于问题 1 和问题 3，前往本地修改并重新覆盖上传系统，文件自动安全升级至系统 v2 版。
*   **预期成功标准**：
    *   忽略历史足迹及理由被完好附带记录并能在后续审核流中呈现。
    *   再次自检无未处理的核心异常，一致性自检结果重归合规态，系统解锁“提交送审”按钮。

## 场景五：审核领导针对技术方案批注与审核意见关闭 (Review-Closure)

*   **测试前置条件**：[施工技术-陈七] 提交的任务文件已完成一致性自检并正式打包送审。使用账号：[审核领导-钱八]、[施工技术-陈七]。
*   **验收测试流程**：
    1.  [审核领导-钱八] 登录工作台，在“我的待办审查”列表中查阅由陈七提交的施工方案。
    2.  点击阅读，在系统中可以直接审阅该方案，并看到与之绑定的“一致性自检报告”（包含由于专家优化被陈七主动忽略的 380 天工期标注）。
    3.  [钱八] 对文件内容提出两条结构化的审核批注：
        *   `批注 1`：现场大门临设选址不当，需修正至基地西侧。状态：`Active`。
        *   `批注 2`：平面布置附图缺失重型吊车停机位。状态：`Active`。
    4.  [施工技术-陈七] 收到审核工作任务待办，查阅到该意见。
    5.  [陈七] 在本地修正大纲后再次覆盖上传提交，系统自动命名该工件为 v3，并在 `批注 1` 与 `批注 2` 下面回复答辩意见。
    6.  [钱八] 刷新审核页，自动对比前一版本更新。点击确认，并将上述两条审核意见手动点击“关闭（Resolved）”。
*   **预期成功标准**：
    *   系统保存完整的审核生命周期操作痕迹，意见状态安全流转为 `Resolved`。
    *   由资料汇总员或负责人检验该文档下的所有审核意见已全数“关闭”后，允许将该文件标记为“最终版”冻结。

## 场景六：项目涉密报价文件 AI 隔离防护 (Zero-AI-Isolation)

*   **测试前置条件**：项目已创建，使用账号：[系统管理员-甲]、[项目普通设计员-王五]、[概算员-赵六]、[报价决策人-宋九]。
*   **验收测试流程**：
    1.  [赵六] 登录项目，在概算编制子计划下拖入上传预算测算底表（系统根据任务安全定义自动设置 `isSensitive = true` 且 `isAiAllowed = false`）。
    2.  [普通设计员-王五] 试图登录项目并在文件共享仓内查看与下载该表格。
    3.  系统级别拦截生效，页面显示警告：“该技术/商务资产为敏感属性，你无查看下载授权”。
    4.  系统向 AI 文卷提取与大盘向量化索引脚本发出服务调拨请求，意图对其做大项目跨大纲问答或 RAG 处理。
*   **预期成功标准**：
    *   安全隔离屏障拦截，凡是标记为 `isAiAllowed == false` 的高敏路径文件，底层 AI 服务网关直接切断其物理读取权限，不可被向量化、不可切片分类、不暴露给外部 API 推理及大模型。
    *   在 IT 系统管理员的系统高阶审计日志中，全盘、忠实记录该试图非授权下载和非受控 AI 向量捕获的打回日志。
