# BidWorks 投标一体化生产平台 MVP 风险评估与消减对策 (Risks)

建筑总承包投标是大额资产的严肃商务博弈，由于引入了一定维度的 AI 自动提取与自动一致性自检，系统在设计和落地中必须防范技术及管理风险，并配备相应的防线消减对策。

## 2. 风险细化与消减隔离底线说明

### 风险 2.1: AI “幻觉 (Hallucination)” 导致重大漏标或报价罚则错判
*   **深层危害**：AI 在提取特殊索赔罚金（如逾期日罚十万误写为一万）或关键日期发生漂移，若项目人员由于过度信任系统而放任自流，可能遭遇巨额商业索赔或直接废标。
*   **消减防御红线**：
    1.  **引证页码强标记（Traceability）**：所有 AI 提炼出的段落均带有原文标签，提供可视化一键直跳高亮，严禁让 AI 生成任何没有原文出处支撑的结论。
    2.  **人工二次确库（Review and Override）**：AI 所析数据在被正式写入数据库前，仅挂载为临时“建议状态”，系统强制相关业务主管在人工审校中点击确认键。

### 风险 2.2: 扫描件图像 PDF 或低质量格式解析失效
*   **深层危害**：对于不包含文本图层的纯图像型 PDF 招标文件，文字切分服务可能发生解析率大幅跌落、甚至解析出错，导致无法生成自动解析和自检。
*   **消减防御红线**：
    1.  **降级退路方案（Fallback Approach）**：系统允许营业员在任何时候选择“无标书手工速创项目”，先将已掌握的核心指标手工填入。
    2.  **手动指标校正（Manual Correction）**：即便自动解析失败，系统也提供便利的手工录入和校准，保障后续倒排计划算法不至于卡死。

### 风险 2.3: 概算、最高让步让利敏感大底案泄露，或被外部公有模型索引
*   **深层危害**：企业最核心和机密的商业底线数据（如真实预算底稿、报价底盘定价）在上传、分析时，如果传递到了不受信任的公有大语言模型接口，可能引发商业机密泄露风险。
*   **消减防御红线**：
    *   **机密安全物理隔离（Secure Isolated File Path）**：所有判定为涉密的文件上载时默认标识 `isSensitive = true` 且 `isAiAllowed = false`。
    *   **AI 读取链路切断机制**：平台的文件服务器和检索服务一旦在请求时校验检测到 `isAiAllowed == false`，直接从数据流和分片中切断，不建索引、不向量化，更不把这部分敏感数据传递进任何外部 AI 模型的 Prompt 之中。

### 风险 2.4: 数据频繁联动引发一线人员“预警疲劳 (Alert Fatigue)”而反感
*   **深层危害**：若高频度修改轻微非主数据字段，就对下游所有已完成的工作台进行频繁的复核要求与高扰动状态转换，一线编制人员会产生厌恶心理，最后为图省事，不经核对就闭眼直接点击“确认已复核、没有任何变化”。
*   **消减防御红线**：
    1.  **精细化中高风险区分处理**：低风险变动（如项目别名小修）绝不硬性将任务文档置为“需重新复核”状态，仅记录在变更日志中供自查。
    2.  **复核一键式对比**：提供简便的指标前后版本变化一键式对照，减少一线人员手工排查两版差异的耗时。

### 风险 2.5: 一线员工认为平台增加了重复录入的行政包袱，抗拒使用
*   **深层危害**：如果用户觉得仅仅是给管理层上传文件以供监控，平台会沦为毫无活力的低频工具，用户可能在截止日前敷衍上传，导致开发数据失实。
*   **消减防御红线**：
    *   **上传即见即所得（Instant Helper Utility）**：确保用户上传文稿后，能快速得到有价值的实效工具反馈（例如多岗位指标一致性冲突自检、历史模板旧地名未脱敏残留检测等）。通过这些切实减轻其工作压力、防止低级废标失误的实用功能，反哺并调动一线人员的积极性。
