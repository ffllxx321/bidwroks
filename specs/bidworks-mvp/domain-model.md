# BidWorks 投标一体化生产平台 MVP 领域模型 (Domain Model)

本规格定义 MVP 核心业务实体的模型设计、关联映射（ER 结构）及实体状态声明，为后续数据库架构提供理论参照。

## 1. 业务实体逻辑关系图 (ER Relations)

投标涉及的业务实体在平台中存在强关联，项目主数据是所有协作实体的根基：

## 2. 关键实体明细设计

### 2.1 Project (项目实体)
* **概念说明**：投标项目的最高层空间载体。任何招标文件、参与人员、倒排计划、自检与审核均隶属于一个特定的项目。
* **主要关联**：
  * 与 `ProjectMembership` 呈 一对多 (1:N)
  * 与 `ProjectMasterData` 呈 一对一宿主 (1:1)
  * 与 `BidTask` 呈 一对多 (1:N)
  * 与 `AuditLog` 呈 一对多 (1:N)

### 2.2 ProjectMasterData (项目主数据)
* **概念说明**：项目的指标信息数据库主板。一个项目有且仅有一组主数据指标，下游各专业方案必须从此处读取数据源（详见 `data-fields.md` ）。
* **主要关联**：
  * 隶属于 `Project` (1:1)
  * 拥有一组字段变更记录 `MasterDataChangeLog` (1:N)
  * 触发多个岗位的影响通知记录 `ImpactNotification` (1:N)

### 2.3 BidTask (投标计划/任务)
* **概念说明**：投标项目中的生产工步任务单元（如临设设计、概算初稿、钢结构技术要案等）。
* **主要关联**：
  * 隶属于 `Project` (N:1)
  * 与负责人 `User (Assignee)` 关联 (N:1)
  * 与审核人 `User (Reviewer)` 关联 (N:1)
  * 挂接产生一组迭代文件 `Document` (1:N)
  * 依赖前置任务 `BidTask` 呈自关联

### 2.4 Document (文件工件)
* **概念说明**：一线各岗在任务作业中上传产出的 Word, PDF 或 Excel。平台通过它驱动版本、自检与审核周期。
* **主要关联**：
  * 隶属于 `BidTask` (N:1)
  * 上传者 `User` (N:1)
  * 拥有多历史版本实体 `FileVersionHistory` (1:N)
  * 触底自检报错明细 `PreCheckResult` (1:N)
  * 拥有结构化批注审核意见 `StructuredReviewComment` (1:N)

### 2.5 StructuredReviewComment (结构化审核意见)
* **概念说明**：审核人在线对某文件（某特定版本下）针对性章节或内容提出的修正指令意见包。
* **主要关联**：
  * 隶属于 `Document` (N:1)
  * 审核发起人 `User` (N:1)
  * 修改解决人 `User` (N:1)

### 2.6 PreCheckResult (上传合规自检结果)
* **概念说明**：文件提交前由一致性自检引擎捕获的越界或文不对题结果。
* **主要关联**：
  * 隶属于 `Document` (1:N)

### 2.7 MasterDataChangeLog & ImpactNotification (主数据字段更迭与关联通知)
* **概念说明**：主数据发生更改时，系统以此表跟踪更迭轨迹，同步派生派发影响通知记录给目标角色岗位。
* **主要关联**：
  * 隶属于 `ProjectMasterData` (N:1)
  * 推送至角色受众 (N:1)

### 2.8 AuditLog (高阶审计日志)
* **概念说明**：管理员和项目负责人审计业务可追溯底表。
* **主要关联**：
  * 关联操作主体 `User` (N:1)
  * 关联项目 `Project` (N:1)
  * 关联资产 `Document` / `ProjectMasterData` (N:1)
